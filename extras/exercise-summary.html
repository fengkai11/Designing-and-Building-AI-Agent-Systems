从零构建 AI Agent 系统

全书练习汇总

本文件汇总第 1–20 章中的练习题，适合用于阶段复盘、训练营作业、读书会讨论，也适合交给 Codex/Claude Code 生成练习项目脚手架。建议不要一次性完成所有练习，而是按照“理解型练习 → 设计型练习 → 实现型练习 → 复盘型练习”的顺序推进。

一、使用方式

  1. 每读完一章，先完成本章练习中的 1–2 个核心题。
  2. 每完成一篇，选择一个综合题做成 Markdown 方案。
  3. 到第 10 章后，开始把练习和代码项目结合起来。
  4. 到第 18–19 章后，至少选择一个真实项目做完整闭环。

第 1 章：为什么要系统学习 Agent

练习 1：区分 Chatbot、Tool Calling 和 Agent

请判断下面任务更适合 Chatbot、Tool Calling，还是 Agent，并说明原因。

  1. 用户问：“什么是 Agent Loop？”
  2. 用户问：“我这个订单现在物流到哪里了？”
  3. 用户说：“帮我持续跟进 30 个潜在客户，根据回复生成下一步动作。”
  4. 用户说：“把 300 美元换算成人民币。”
  5. 用户说：“阅读这个代码仓库，找出登录模块 bug，并提交修改建议。”
  6. 用户说：“每天早上帮我汇总行业新闻并标记对我业务有影响的内容。”

重点不是答案唯一，而是说明判断依据。

练习 2：分析一个任务是否适合 Agent

选择你熟悉的一个真实任务，例如拼多多选品、外贸客户开发、学生错题管理、代码 bug 修复、机器人数据采集流程管理等。

请回答：

  1. 这个任务的最终目标是什么？
  2. 它是否需要多步骤执行？
  3. 中间是否存在不确定判断？
  4. 是否需要调用工具？
  5. 是否需要长期记忆？
  6. 哪些动作有风险？
  7. 哪些环节必须人工确认？
  8. 如果不用 Agent，用普通工作流能否完成？

练习 3：找出 Demo 到产品的缺口

设想你做了一个 Demo：输入产品名称和国家，Agent 自动搜索客户并生成客户列表。

请列出至少 10 个真实上线后会遇到的问题，例如重复客户、错误客户、联系方式无效、搜索结果质量差、邮件风险等。然后为每个问题写出可能需要的系统机制。

示例：

问题：同一家公司被多次找到。
机制：客户去重、域名规范化、公司名称相似度匹配。

练习 4：设计一个安全边界

假设你正在设计一个代码开发 Agent，它可以读取文件、修改文件、运行测试、执行 shell 命令。

请把工具分成三类：

  1. 可以自动执行；
  2. 执行前需要用户确认；
  3. 永远禁止执行。

并说明每类的理由。

练习 5：识别长期记忆

假设你在设计一个教育 Agent，用于帮助老师管理学生学习。

请判断下面哪些信息应该进入长期记忆，哪些只应该保留在当前任务中，哪些不应该记录。

  1. 学生最近 10 次错题的知识点；
  2. 学生今天说“我现在有点累”；
  3. 老师偏好每周五查看学习报告；
  4. 某学生反复在几何证明中漏写理由；
  5. 一次临时测试的草稿答案；
  6. 老师要求“以后报告用简洁格式”；
  7. 学生的敏感私人信息。

第 2 章：Agent 的能力模型

练习 1：为一个任务建立能力模型

选择一个你熟悉的任务，例如外贸客户开发、拼多多选品、代码 bug 修复、学生错题管理、机器人数据采集管理。

请按照十层能力模型分别回答：

  1. 目标、任务与意图理解层：用户目标是什么？需要哪些任务参数？成功标准是什么？
  2. 观察、感知与信息获取层：Agent 需要从哪里获取信息？哪些信息可信？哪些信息需要验证？
  3. 规划与控制层：任务适合 ReAct、Plan-and-Execute，还是工作流 + Agent？
  4. 工具与行动层：需要哪些工具？哪些工具低风险？哪些工具高风险？
  5. 上下文工程层：每一步模型需要看到哪些信息？哪些信息不应该进入上下文？
  6. 记忆与知识层：哪些信息应该短期保存？哪些应该进入长期记忆？
  7. 状态、调度与长期任务层：任务是否需要跨天执行？是否需要 checkpoint？
  8. 评估、反思与可观测层：如何判断结果好坏？需要记录哪些执行轨迹？
  9. 安全、权限与治理层：哪些动作必须审批？哪些动作应该禁止？
  10. 产品交互与人机协同层：用户如何输入、查看、纠正和审批？

练习 2：判断缺失能力层

假设有一个 Agent Demo，可以根据用户输入自动搜索网页并生成客户列表。但是它存在以下问题：

  1. 经常找到重复公司；
  2. 不知道哪些客户已经联系过；
  3. 有时把零售店当成批发商；
  4. 搜索过程无法回放；
  5. 生成邮件后直接发送；
  6. 用户不知道任务执行到哪一步；
  7. 搜索失败后从头开始。

请判断每个问题主要对应能力模型中的哪一层缺失，并提出改进机制。

示例：

问题：经常找到重复公司。
对应能力层：记忆与知识层 / 状态层。
改进机制：客户去重表、域名规范化、历史线索记忆。

练习 3：设计工具权限分级

为一个代码开发 Agent 设计工具权限表，至少包含以下工具：

  • 读取文件；
  • 搜索代码；
  • 写入文件；
  • 运行测试；
  • 执行 shell 命令；
  • 删除文件；
  • 提交 git commit；
  • 推送远程仓库。

请把它们分成：

  1. 可自动执行；
  2. 需要用户确认；
  3. 默认禁止。

并说明理由。

练习 4：设计上下文

假设外贸 Agent 正在判断一家公司是否适合成为钢卷尺客户。请写出本次模型调用应该看到的上下文结构。

至少包括：

同时说明哪些信息不应该放入本次上下文。

练习 5：用能力模型分析一个产品

选择一个你熟悉的 Agent 产品或开源项目，例如 Claude Code、Codex、OpenClaw、某个浏览器 Agent、某个客服 Agent。

不需要知道其内部源码，只从外部行为出发，按照十层能力模型分析：

第 3 章：什么时候该用 Agent，什么时候不该用 Agent

练习 1：判断自动化形态

  1. 根据售价、成本和推广费计算商品毛利。
  2. 每天凌晨导出订单数据并生成销量报表。

第 4 章：Agent Loop：从 Observe 到 Act

练习 1：画出一个 Agent Loop

选择一个你熟悉的任务，例如“拼多多选品分析”“外贸客户开发”“代码 bug 修复”“学生错题复习”。

请画出它的 Agent Loop，至少包含：

  1. Observe：Agent 需要观察什么信息？
  2. Think：Agent 需要判断什么？
  3. Plan：是否需要多步计划？
  4. Act：需要调用哪些工具？
  5. Reflect：什么情况下需要调整策略？
  6. Stop：什么时候停止？

重点不是画得漂亮，而是把任务推进过程拆清楚。

练习 2：为外贸 Agent 设计停止条件

假设你要做一个外贸客户开发 Agent，任务是“每天寻找 10 个适合钢卷尺的潜在客户”。

请设计至少 6 类停止条件，包括：

  1. 目标完成；
  2. 搜索次数上限；
  3. 时间上限；
  4. 连续失败；
  5. 低质量结果；
  6. 等待人工审批。

并说明每个停止条件触发后，系统应该如何向用户汇报。

练习 3：识别循环失控

  1. 这个 Agent 出现了什么问题？

  3. 你会如何要求 Agent 更新搜索策略？

练习 4：实现一个最小 Agent Loop

练习 5：加入人工确认状态

在练习 4 的基础上，增加一种动作类型：need_approval。

当 Agent 准备执行高风险操作时，例如“发送邮件”或“删除文件”，Runner 不应该继续执行，而应该把任务状态设置为 waiting_for_user。

  2. 为一个外贸客户开发任务画出 Agent 能力模型。
  3. 实现一个最小 Agent Loop，并增加最大步数限制。
  4. 设计一个安全的 Tool Registry，区分只读工具、写入工具和高风险工具。
  5. 为一个真实业务任务设计上下文注入策略。
  6. 比较 ReAct 与 Planner-Executor 在同一任务中的差异。
  7. 设计 Memory Store 的 schema，并说明哪些内容不应写入长期记忆。
  8. 实现一个最小 Markdown RAG 检索流程。
  9. 设计长任务状态机，支持暂停、恢复、取消和重试。
  10. 为邮件发送、代码修改、数据库写入设计审批机制。
  11. 为外贸客户开发 Agent 构造 20 条评估任务。
  12. 选择一个综合项目，完成端到端设计文档和最小代码原型。

三、练习评分建议

每个练习可以按 5 个维度评分：

维度说明
问题理解是否准确识别任务目标和边界
机制完整性是否覆盖工具、上下文、状态、记忆、安全等关键机制
工程可行性是否能落到代码、数据结构、接口和运行流程
风险控制是否识别高风险动作并设置审批或禁止策略
可复盘性是否留下日志、指标、检查清单和改进入口
