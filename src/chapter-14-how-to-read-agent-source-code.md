# 第 14 章：如何阅读 Agent 项目源码

> 从源码到架构抽象，建议先看映射图，再带着“项目—模式—Runtime 模块”的思路进入正文。

![图 14-16-1 源码研究到 Agent Runtime 的映射图](figures/fig-14-16-01-source-to-runtime-mapping.png)

*图 14-16-1 源码研究到 Agent Runtime 的映射图*


前面十三章，我们已经从系统角度拆解了 Agent 的核心机制：Agent Loop、工具系统、上下文工程、Planning、Memory、RAG、长期任务、人机协同、评估和可观测性。到这一章，读者很容易产生一个自然冲动：既然概念已经学了，接下来是不是应该直接去读成熟项目源码？

答案是：应该读，而且必须读。但不能盲目读。

Agent 项目的源码通常不像一个普通 CRUD 项目那样容易理解。一个看似简单的 Agent 项目，背后可能同时包含模型适配层、工具注册层、上下文组装层、任务状态机、事件流、前端工作台、权限系统、日志系统、插件系统、调度系统和持久化层。你打开仓库时，很可能看到几十个目录、几百个文件、多个 package、多个运行入口、多个环境变量和大量配置。如果没有方法，很容易陷入“每个文件都能看懂一点，但整体仍然不懂”的状态。

很多人读源码失败，不是因为代码能力不够，而是因为读法错了。他们一上来就从 `main.py`、`index.ts` 或 `app.ts` 开始逐行看，试图靠线性阅读理解系统。这种方式适合小脚本，却不适合 Agent 系统。Agent 项目的核心不在某一个函数里，而在一条运行链路里：用户目标如何进入系统，如何变成任务，任务如何进入 runtime，runtime 如何组装上下文，模型如何选择工具，工具结果如何写回状态，状态如何驱动下一步，风险动作如何进入审批，最终结果如何展示给用户。

因此，阅读 Agent 项目源码的第一原则是：

> 不要先读文件，先读链路。

本章要解决的问题，就是给读者一套可复用的源码阅读方法。以后无论你研究 OpenClaw、Claude Code、Codex CLI、LangGraph 应用、CrewAI 项目，还是你自己团队写的 Agent 系统，都可以用这套方法快速建立整体地图。

---

## 14.1 为什么 Agent 源码难读

Agent 源码难读，并不是因为每一行代码都很深奥。很多时候，单个函数并不复杂，难的是多个模块之间的关系。

传统 Web 项目的阅读路径相对明确。比如一个典型后端项目，你通常会先找路由，再找 controller，再找 service，再找 repository，再找数据库模型。请求进来后，大致是沿着同步调用链往下走。即使项目很大，链路也相对稳定。

Agent 系统不同。它经常不是一次请求返回一次响应，而是一次用户目标触发一段持续执行过程。这个过程可能包含异步任务、事件队列、多轮模型调用、工具调用、状态更新、前端推送和人工审批。执行顺序不是固定函数调用，而更像一条动态轨迹。

例如，一个外贸客户开发 Agent 收到任务：

```text
帮我寻找沙特市场中适合钢卷尺出口的五金批发商，优先找有进口能力的公司，排除零售小店，最后生成 20 个客户线索和开发信草稿。
```

源码中可能发生的链路是：

1. API 层接收用户输入；
2. 创建 `Task` 记录；
3. 写入任务队列；
4. Runtime 拉取任务；
5. Planner 生成初始计划；
6. Context Manager 组装系统提示词、用户目标、已有记忆和工具说明；
7. Model Client 调用 LLM；
8. LLM 返回下一步动作；
9. Tool Executor 执行搜索工具；
10. 工具结果写入 Task State；
11. Runtime 再次组装上下文；
12. LLM 决定读取某个网页；
13. 工具返回网页摘要；
14. Agent 判断客户类型；
15. Scoring 模块计算商机分；
16. 邮件草稿进入 Approval Queue；
17. 前端收到事件流，展示进度；
18. 用户审批后触发后续发送或导出。

这条链路可能跨越十几个文件。如果你只看其中一个文件，就会觉得它“好像只是一个工具函数”“好像只是一个 prompt 模板”“好像只是一个 API handler”。但把它们串起来，才是 Agent。

另一个难点是，Agent 项目经常混合确定性代码和非确定性模型调用。普通函数的输入输出比较稳定，而模型调用的输出会变化。源码中真正关键的部分，往往不是算法逻辑，而是约束模型行为的结构：schema、prompt、state、tool contract、approval policy、retry policy、stop condition。这些东西不一定集中在一个地方，可能散落在配置文件、类型定义、prompt 文件、工具声明和运行时逻辑中。

第三个难点是，Agent 项目常常同时包含“产品形态”和“底层机制”。例如一个代码 Agent，表面上是终端命令或 IDE 插件，底层却有 repo indexing、file search、shell sandbox、diff generation、test runner、rollback 等机制。如果你只看 UI 层，会误以为它只是聊天界面；如果只看 runtime 层，又会忽略用户确认和结果展示。

所以，读 Agent 源码需要一种系统化视角：先理解它要完成什么工作，再找到工作流入口，然后沿着任务生命周期追踪，再拆解关键机制。

## 14.2 源码阅读的三个目标

在正式读源码之前，必须先明确你为什么读。

不同目标，对应不同读法。没有目标的源码阅读，很容易变成无效浏览。

第一个目标，是学习产品形态。

你想知道这个 Agent 是怎么被用户使用的：是 CLI、Web、移动端、聊天渠道、IDE 插件，还是后台任务？用户如何输入目标？如何查看执行过程？如何审批高风险动作？如何接收最终结果？

例如，研究一个个人助理型 Agent 时，你可能更关心它如何接入 WhatsApp、Telegram、Slack、iMessage 等渠道，如何把不同渠道消息统一成内部事件，如何把回复返回原渠道。这里重点不是模型调用，而是产品入口和消息路由。

第二个目标，是学习运行时机制。

你想知道 Agent 是如何执行任务的：是否有 Agent Loop？是否有 Planner / Executor 分离？是否有状态机？是否有事件总线？工具如何注册？上下文如何组装？长任务如何恢复？

例如，研究一个代码 Agent 时，你应该重点看它如何读取代码库、如何选择文件、如何调用 shell、如何生成 diff、如何处理测试失败。这里重点是 runtime，而不是 UI。

第三个目标，是学习可复用模块。

你想判断哪些设计可以搬到自己的项目里。比如工具注册系统、审批队列、任务事件日志、记忆写入策略、插件接口、配置机制、评估 harness 等。

这三个目标经常同时存在，但优先级不同。如果你是为了做自己的 Agent Runtime，重点应该放在第二和第三个目标；如果你是为了做产品，第一目标也非常重要。

读源码前可以写一张“阅读目标卡”：

```markdown
# 源码阅读目标卡

项目名称：
项目类型：个人助理 / 代码 Agent / 浏览器 Agent / 企业流程 Agent / 多 Agent 框架

我读这个项目的目的：
- [ ] 学产品形态
- [ ] 学运行时机制
- [ ] 学工具系统
- [ ] 学上下文工程
- [ ] 学长期任务
- [ ] 学安全与审批
- [ ] 学前端交互
- [ ] 学工程组织

本次阅读要回答的 5 个问题：
1.
2.
3.
4.
5.
```

这个小模板很重要。它能阻止你陷入无目的阅读。

## 14.3 第一步：先看产品入口，不要先看模型调用

很多人读 Agent 项目源码时，第一反应是搜索 `openai`、`anthropic`、`model`、`llm`、`chat.completions`。这当然能快速找到模型调用位置，但不建议一开始这么做。

原因是：模型调用只是 Agent 链路中的一个节点。你先看模型调用，很容易误以为 Agent 的核心就是“发一个 prompt 给模型”。但真实系统的核心问题往往发生在模型调用之前和之后。

更好的第一步，是先看产品入口。

产品入口回答的是：用户从哪里进入系统？

一个 Agent 项目常见的入口包括：

```text
CLI 命令入口：用户在终端输入任务。
Web API 入口：前端通过 HTTP/WebSocket 提交任务。
聊天渠道入口：用户通过 Telegram、Slack、微信、邮件等发送消息。
IDE 插件入口：用户在编辑器里发起代码任务。
定时任务入口：系统按计划自动触发任务。
Webhook 入口：外部系统事件触发 Agent。
```

不同入口会影响整个架构。

如果入口是 CLI，系统通常更偏本地运行，工具权限可能包括文件系统和 shell，交互方式偏流式日志和终端确认。

如果入口是 Web，系统通常需要任务持久化、用户会话、权限、前端状态同步和多用户隔离。

如果入口是聊天渠道，系统需要处理消息去重、渠道适配、用户身份映射、异步回复和多端状态。

如果入口是定时任务，系统需要 scheduler、任务恢复、重复执行控制和通知机制。

例如，一个个人助理型 Agent，如果支持多个聊天渠道，它的源码里一定会有 channel adapter 或 connector。阅读时应该先找：不同渠道消息如何被标准化？它们是否被统一转换成 `MessageEvent`？用户身份如何映射？消息是否进入统一队列？

一个代码 Agent，如果是 CLI 工具，阅读时应该先找命令入口：命令参数如何解析？当前工作目录如何确定？repo context 如何初始化？用户输入如何进入 agent session？

入口层决定了 Agent 的“使用场景”。理解入口后，再看 runtime 才不会迷路。

## 14.4 第二步：画出任务生命周期

找到入口后，下一步不是继续看实现细节，而是画任务生命周期。

任务生命周期回答的是：一个用户目标从进入系统到结束，中间经历哪些状态？

一个简单 Agent 可能只有三种状态：

```text
created -> running -> completed / failed
```

但真实 Agent 往往更复杂：

```text
created
-> planning
-> waiting_for_tool
-> processing_tool_result
-> waiting_for_approval
-> running_after_approval
-> completed
-> failed
-> cancelled
-> paused
-> resumed
```

为什么任务生命周期这么重要？因为它是阅读源码的主线。

你可以把源码中所有模块都挂到生命周期上：

- API 层负责创建任务；
- Scheduler 负责启动任务；
- Runtime 负责 running；
- Planner 负责 planning；
- Tool Executor 负责 waiting_for_tool 和 processing_tool_result；
- Approval Queue 负责 waiting_for_approval；
- Event Bus 负责把状态变化推送给前端；
- Store 负责持久化状态；
- Eval 模块负责完成后的质量判断。

如果没有生命周期图，你看到的只是零散模块；有了生命周期图，你会知道每个模块在什么时候发挥作用。

阅读源码时，可以建立一个表：

```markdown
| 状态 | 触发条件 | 负责模块 | 输入 | 输出 | 失败处理 |
|---|---|---|---|---|---|
| created | 用户提交任务 | API / TaskService | user_goal | task_id | 参数校验失败 |
| planning | runtime 启动 | Planner | task + context | plan | 计划为空则重试 |
| waiting_for_tool | 模型请求工具 | ToolExecutor | tool_call | tool_result | timeout / retry |
| waiting_for_approval | 高风险动作 | ApprovalService | draft/action | approve/reject | 用户拒绝则改写 |
| completed | stop condition 满足 | Runtime | final_state | result | 生成报告失败 |
```

这个表不需要一开始就完整。你可以边读边补。

特别注意：很多项目的状态不一定显式命名为 `TaskState`。有些项目可能用数据库字段、内存对象、事件日志、队列消息或前端 session 来隐式表示状态。你要找的不是某个名字，而是“系统如何知道当前执行到哪一步”。

## 14.5 第三步：追踪 Agent Loop，而不是只看 prompt

任务生命周期确定后，就可以开始追踪 Agent Loop。

Agent Loop 是 Agent 项目的心脏。它负责把“当前状态”变成“下一步行动”。

一个典型 loop 可以抽象为：

```python
while not done:
    observation = collect_observation(state)
    context = build_context(task, state, memory, tools)
    decision = model.generate(context)
    action = parse_decision(decision)
    result = execute_action(action)
    state = update_state(state, action, result)
```

不同项目实现差异很大，但核心问题类似：

1. loop 在哪里启动？
2. 每次循环输入什么？
3. 模型输出格式是什么？
4. 如何解析模型输出？
5. 工具调用如何执行？
6. 工具结果如何写回状态？
7. 如何决定停止？
8. 如何防止无限循环？
9. 失败后如何重试？
10. 是否支持暂停和恢复？

阅读源码时，要重点寻找这些关键词或概念：

```text
run
step
execute
agent_loop
iteration
turn
state
trace
tool_call
observation
action
finish
stop_condition
max_steps
```

但是，不要机械搜索。不同语言和项目命名不同。关键是理解它在做什么。

例如，有些框架把 loop 封装在图执行器里，源码中可能没有显式 `while`。每个节点执行后根据状态转移到下一个节点，本质上仍然是 loop。

有些代码 Agent 可能把一次模型调用称为 `turn`，把工具结果称为 `observation`，把中间记录称为 `transcript`。这些命名不同，但对应机制相同。

追踪 Agent Loop 时，最重要的是找到“状态更新点”。因为 Agent 的每一步都应该改变状态。如果模型调用后没有把结果可靠写入状态，下一步就可能丢失上下文。

例如，Agent 调用了搜索工具，得到 10 个客户网页。如果这些结果只作为一段文本塞进下一次 prompt，而没有结构化保存，那么后续去重、评分、引用来源都会困难。更好的设计是把候选客户写入 `LeadCandidate` 结构，并记录来源 URL、抓取时间、判断依据和置信度。

所以，读 Agent Loop 时，不要只看模型怎么调，要看每一步如何改变系统状态。

## 14.6 第四步：拆解工具系统

工具系统是 Agent 源码中最值得认真读的部分之一。

工具决定了 Agent 能做什么，也决定了它可能造成什么风险。一个项目的工具系统设计，往往能反映作者对 Agent 的理解深度。

阅读工具系统时，不要只看“有哪些工具”，而要看工具是如何被描述、注册、选择、执行和审计的。

一个成熟工具系统至少应该回答：

```text
工具名称是什么？
工具描述是否清晰？
参数 schema 是否严格？
返回值结构是否稳定？
工具是否有权限等级？
工具是否支持 dry-run？
是否有 timeout？
是否有 retry？
是否有幂等性设计？
是否记录调用日志？
是否能被禁用？
高风险工具是否进入审批？
```

以文件工具为例，读源码时应该检查：

- `read_file` 是否限制路径范围？
- `write_file` 是否会覆盖文件？
- 写入前是否生成 diff？
- 是否支持备份或 checkpoint？
- 是否禁止访问 `.env`、密钥、系统目录？
- 是否把完整文件内容暴露给模型？

以 shell 工具为例，应该检查：

- 是否有命令白名单或黑名单？
- 是否限制工作目录？
- 是否限制执行时间？
- 是否限制网络访问？
- 是否截断输出？
- 是否对危险命令二次确认？
- 是否记录 stdout、stderr、exit code？

以邮件工具为例，应该检查：

- 是否直接发送邮件？
- 是否只生成草稿？
- 是否需要人工审批？
- 是否记录收件人、主题、正文、附件？
- 是否有频率限制？
- 是否防止重复触达？

一个常见的坏设计是把底层 API 原封不动暴露给模型。例如把 `send_email(to, subject, body)` 直接作为工具交给 Agent。这种设计简单，但风险很高。更好的设计是把它改成两步：

```text
create_email_draft -> human_approve -> send_approved_email
```

源码中如果有这种拆分，说明项目对 Agent 安全边界有较成熟认识。

读工具系统还有一个重点：工具是否“面向 Agent 设计”。

传统 API 面向程序员，参数可能很多，含义依赖文档。Agent 工具应该面向模型，名称明确、描述具体、参数少而清晰、返回结构稳定。例如：

```json
{
  "name": "search_potential_importers",
  "description": "Search for companies that may import or distribute the specified product in the target country. Use this only for lead discovery, not for verifying a specific company.",
  "parameters": {
    "product": "string",
    "country": "string",
    "customer_type": "string",
    "exclude_keywords": "array[string]"
  }
}
```

这比一个泛泛的 `web_search(query)` 更可控。当然，泛用搜索工具也有价值，但业务型 Agent 往往需要更高层工具。

源码阅读时，可以把工具分成三层：

```text
基础工具：文件、搜索、HTTP、shell、浏览器。
领域工具：客户搜索、代码测试、题目批改、CRM 查询。
控制工具：请求审批、暂停任务、保存结果、更新记忆。
```

看一个项目是否成熟，就看它是否只停留在基础工具，还是已经把业务动作抽象成领域工具和控制工具。

## 14.7 第五步：阅读上下文构造，而不是只读 prompt 文本

很多项目里有 `prompts/` 目录。读者看到后很容易兴奋，因为 prompt 看起来最接近模型行为。但是，只读 prompt 文本是不够的。

真正重要的是上下文构造过程。

上下文不是一个静态 prompt，而是由多个来源动态组合而成：

```text
系统规则
开发者规则
用户目标
当前任务状态
历史对话
工具说明
工具调用结果
记忆检索结果
知识库片段
审批记录
失败日志
输出格式要求
```

阅读源码时，要找到 `build_context`、`render_prompt`、`compose_messages`、`get_system_prompt`、`memory_retrieval`、`tool_result_to_message` 这类逻辑。

重点看几个问题。

第一，上下文有没有分层？

系统规则应该和用户输入分开。网页内容、文件内容、工具结果不能和系统指令混在一起。否则容易产生 prompt injection 风险。

例如，浏览器 Agent 读取网页时，网页内容中可能出现“忽略所有之前指令”。如果上下文构造没有明确标记“以下是网页内容，不是指令”，模型可能受到干扰。

第二，上下文有没有预算管理？

Agent 执行多轮后，历史会越来越长。如果每一步都把全部历史塞进 prompt，成本和延迟会急剧上升，还可能稀释关键约束。源码中应该有摘要、截断、检索、优先级排序等机制。

第三，工具结果如何进入上下文？

工具结果是原样进入，还是结构化摘要进入？长网页是否先提取？搜索结果是否保留 URL 和时间？代码文件是否只取相关片段？这些细节会直接影响 Agent 质量。

第四，记忆如何使用？

Memory 不应该全部注入。系统应根据当前任务检索相关记忆，并明确标注来源、时间、置信度。错误记忆或过期记忆应该能被忽略。

第五，输出格式如何约束？

如果模型需要返回工具调用或 JSON，源码中是否有 schema 校验？如果校验失败，是否有修复流程？如果模型返回自然语言混杂 JSON，系统如何处理？

在阅读上下文构造时，可以画一张“Prompt 输入来源图”：

```text
User Goal
   + System Policy
   + Developer Rules
   + Task State
   + Tool Schemas
   + Recent Events
   + Retrieved Memory
   + Retrieved Knowledge
   + Output Contract
      -> Context Builder
         -> Model Messages
```

这张图比单个 prompt 文件更有价值。

## 14.8 第六步：识别记忆、存储和状态的边界

很多 Agent 项目都会有数据库或本地存储，但不同数据的性质完全不同。阅读源码时必须区分三类东西：状态、记忆和知识库。

状态是当前任务执行过程中的数据。例如任务目标、当前步骤、已调用工具、候选客户列表、待审批邮件、错误信息、执行进度。状态服务于“完成当前任务”。

记忆是跨任务复用的信息。例如用户偏好、历史客户触达记录、学生长期错因、代码项目经验、成功策略。记忆服务于“未来任务做得更好”。

知识库是外部领域知识。例如产品规格文档、公司介绍、API 文档、课程教材、业务规则。知识库服务于“回答和执行时有依据”。

三者不能混淆。

如果把任务状态当长期记忆，会污染未来任务。例如某次任务临时要求“这次邮件语气夸张一点”，不应该永久影响所有开发信。

如果把长期记忆只放在上下文里，不持久化，就无法跨任务复用。

如果把知识库当记忆，系统可能无法区分“事实资料”和“用户偏好”。

源码阅读时，要关注项目如何存储这些数据：

```text
任务状态：数据库表、JSON 文件、事件日志、队列消息。
长期记忆：profile 表、memory store、向量库、结构化偏好。
知识库：文档索引、embedding store、文件系统、外部数据库。
```

也要看数据是否有生命周期。

任务状态完成后是否归档？长期记忆是否可更新、删除、纠错？知识库是否支持版本？如果没有这些机制，Agent 长期运行后会越来越混乱。

一个好的源码阅读问题是：

> 如果这个 Agent 连续运行三个月，它的数据会变成资产，还是变成垃圾？

这句话非常实用。很多 Demo 项目短期能跑，但没有数据治理能力。真正的 Agent 产品必须考虑长期状态和记忆治理。

---

## 14.9 第七步：看安全边界和审批机制

Agent 项目最容易被忽视、也最能体现成熟度的部分，是安全边界。

读源码时，要主动寻找以下机制：

```text
权限模型
工具风险等级
人工审批
沙箱
路径限制
网络限制
密钥保护
敏感信息过滤
危险命令拦截
速率限制
审计日志
回滚机制
```

如果一个 Agent 项目把所有工具都直接交给模型，没有权限区分，没有审批，没有日志，那么它最多适合作为实验项目，不适合真实使用。

安全边界不是后期再加的小功能，而是 Agent 架构的一部分。

例如，代码 Agent 的 shell 工具如果没有安全策略，就可能执行危险命令。合理设计应该至少包括：

```text
默认只在项目目录内执行；
禁止访问用户主目录敏感文件；
限制命令执行时间；
高风险命令需要确认；
网络访问可配置；
输出截断并脱敏；
所有命令记录到 trace；
修改文件前创建 checkpoint。
```

外贸 Agent 的邮件工具也类似：

```text
默认只能生成草稿；
发送前必须审批；
同一客户不能重复触达；
退订和拒绝必须进入黑名单；
邮件内容必须保留产品事实依据；
批量发送必须有频率限制；
联系人来源必须可追溯。
```

读源码时，不要只看功能是否能跑，要看坏情况如何处理。

一个成熟项目往往会在“失败路径”上花很多代码。比如工具超时怎么办？模型输出格式错怎么办？审批被拒怎么办？任务被取消怎么办？网络断了怎么办？用户中途修改目标怎么办？

这些代码不如 Demo 炫酷，但它们决定系统是否能上线。

---

## 14.10 第八步：看可观测性和调试体验

Agent 的行为具有不确定性，因此可观测性非常关键。

如果一个 Agent 失败了，你必须能回答：

```text
它看到了什么？
它为什么选择这一步？
它调用了哪个工具？
工具返回了什么？
状态如何变化？
哪一步开始偏离？
花了多少 token？
耗时多久？
是否触发过重试？
最终为什么停止？
```

源码中对应的机制包括 trace、event log、span、tool call log、model request log、cost tracking、replay、debug UI 等。

阅读时可以检查：

- 是否记录每一轮模型输入输出？
- 是否记录工具调用参数和结果？
- 是否记录状态变化？
- 是否记录用户审批？
- 是否可以回放执行过程？
- 是否可以导出日志用于评估？
- 是否能定位成本和延迟？

没有可观测性的 Agent，很难改进。因为你不知道它为什么成功，也不知道它为什么失败。

一个好的 Agent 项目，其实应该像一个实验系统。每次执行都是一次可复盘实验。执行轨迹就是训练材料、评估材料和调试材料。

源码中如果有 `trace_id`、`run_id`、`step_id`、`event_type` 这些概念，通常值得重点阅读。

---

## 14.11 第九步：看前端或交互层

很多技术开发者读源码时容易忽略前端。但 Agent 项目的前端不只是展示结果，它是人机协同的一部分。

一个真实 Agent 系统，用户需要知道：

```text
任务现在执行到哪一步；
Agent 已经找到了什么；
下一步准备做什么；
哪些动作需要我确认；
哪些结果可信，哪些不确定；
我能否修改目标；
我能否暂停、恢复或取消任务；
我能否查看历史记录；
我能否纠正错误记忆。
```

这些都属于产品交互层。

例如，外贸 Agent 如果只在终端输出一堆日志，业务用户很难使用。更合理的界面应该有候选客户表、商机评分、来源链接、开发信草稿、审批按钮、触达状态和跟进提醒。

代码 Agent 的交互也不只是聊天框。它需要展示修改计划、文件 diff、测试结果、失败日志、待确认命令和回滚选项。

因此，读 Agent 源码时要看 UI 如何映射 runtime 状态。前端显示的每个状态，背后都应该有后端事件或任务状态支持。

如果前端只展示最终答案，说明系统更像 Chatbot；如果前端能展示过程、审批、历史和控制，说明它更接近真实 Agent 工作台。

---

## 14.12 第十步：把源码转化为自己的架构素材

读源码的最终目的，不是赞叹别人写得好，也不是机械复制目录结构，而是把它转化为自己的架构素材。

每读完一个项目，都应该输出三份东西。

第一份是运行链路图。

```text
User Input -> API/CLI -> Task -> Runtime -> Context -> Model -> Tool -> State -> UI/Result
```

你要用自己的话画出来，不能只复制项目文档。

第二份是模块职责表。

```markdown
| 模块 | 职责 | 可复用价值 | 复杂度 | 是否适合搬到我的项目 |
|---|---|---|---|---|
| Tool Registry | 工具注册与 schema 暴露 | 高 | 中 | 是 |
| Approval Queue | 高风险动作审批 | 高 | 中 | 是 |
| Channel Adapter | 多聊天渠道接入 | 中 | 高 | 视场景而定 |
```

第三份是设计取舍总结。

你要回答：这个项目为什么这样设计？它解决了什么问题？代价是什么？如果放到我的场景，需要修改什么？

例如，一个个人助理项目支持很多聊天渠道，这对个人助手很有价值，但对外贸 Agent 的第一版可能不是重点。相反，它的消息统一、事件路由和长期运行机制可能值得借鉴。

一个代码 Agent 的 shell sandbox 对代码项目非常关键，但对教育 Agent 可能不需要。它的 diff 展示和用户确认机制，则可以迁移到“AI 自动生成学习计划后老师审批”的场景。

源码阅读真正有效的标志，不是你看懂多少文件，而是你能说清楚哪些机制值得学、哪些机制暂时不用、哪些机制需要改造。

## 14.13 一个源码阅读示例：代码 Agent

假设你要阅读一个代码 Agent 项目。可以按下面路径进行。

第一步，看入口。

它是 CLI 还是 IDE 插件？用户是在终端输入 `agent "fix login bug"`，还是在 VS Code 里点击按钮？入口决定它如何获取当前项目目录、用户身份和交互方式。

第二步，看 repo context。

它如何理解代码库？是否扫描文件树？是否读取 README？是否建立索引？是否识别语言和框架？是否忽略 `node_modules`、`.git`、构建产物？

第三步，看任务计划。

它是否先生成计划？计划是否展示给用户？用户能否修改？计划是否结构化？

第四步，看文件工具。

它如何读文件、写文件、搜索文本、生成 diff？是否限制路径？是否支持 rollback？

第五步，看 shell 工具。

它如何运行测试？命令如何审批？输出如何截断？失败如何反馈给模型？

第六步，看 loop。

测试失败后，系统如何把错误日志重新注入上下文？模型如何决定修改哪个文件？是否有最大修复次数？

第七步，看最终输出。

它是否生成变更摘要？是否展示 diff？是否建议 commit message？是否让用户 review？

这条路径比从第一个文件逐行读更高效。

## 14.14 一个源码阅读示例：外贸客户开发 Agent

如果你阅读的是一个业务型 Agent，比如外贸客户开发 Agent，可以按另一条路径。

第一步，看任务输入。

用户输入哪些字段？产品、国家、客户类型、禁止触达名单、价格区间、邮件风格、工厂能力是否结构化？如果这些信息只是自然语言，后续执行会更不稳定。

第二步，看搜索策略。

Agent 如何生成关键词？是否支持多语言？是否区分进口商、分销商、零售商？是否记录搜索 query？

第三步，看客户识别。

系统如何判断一个网站对应的公司是否是目标客户？是靠模型分类、规则、网页信号，还是人工确认？是否保留判断依据？

第四步，看去重。

是否按域名、公司名、邮箱、地址去重？是否处理同一公司多个网站或多个分支？

第五步，看评分。

评分维度是什么？是否包括匹配度、规模信号、联系方式质量、市场相关性、风险信号？评分是否可解释？

第六步，看邮件草稿。

开发信是否基于真实客户信息生成？是否避免夸大？是否引用产品知识库？是否进入审批队列？

第七步，看长期跟进。

系统是否记录触达历史？是否根据回复调整下一步？是否有黑名单和频率控制？

这条阅读路径能让你快速判断一个外贸 Agent 是“看起来会找客户”，还是“真的能进入业务流程”。

## 练习题

### 练习 1：选择一个 Agent 项目并画链路图

选择一个你感兴趣的 Agent 项目，可以是代码 Agent、浏览器 Agent、个人助理 Agent 或业务 Agent。不要一开始逐行读源码，先回答：

1. 用户从哪里输入任务？
2. 任务如何创建？
3. Agent Loop 在哪里？
4. 工具如何执行？
5. 状态如何保存？
6. 最终结果如何返回？

画出一张运行链路图。

### 练习 2：拆解工具系统

找出项目中的工具定义，列出至少 10 个工具，并按低风险、中风险、高风险分类。说明哪些工具应该自动执行，哪些需要审批，哪些不应开放给 Agent。

### 练习 3：分析上下文构造

找到项目中构造模型上下文的位置。分析上下文包含哪些信息：系统规则、用户目标、工具说明、历史记录、记忆、知识库、工具结果。判断是否存在上下文过长、信息混杂或 prompt injection 风险。

### 练习 4：评估长期运行能力

假设这个 Agent 连续运行三个月，它会遇到什么数据问题？任务历史、记忆、日志、失败记录、用户偏好是否会变成资产？还是会变成难以维护的垃圾？提出改进建议。

使用本章模板，为一个项目写一份不少于 2000 字的源码阅读记录。重点不是复述文件结构，而是解释运行链路、核心机制和可借鉴设计。

## 本章总结

Agent 项目源码的难点，不在于某个函数特别复杂，而在于系统链路长、模块关系多、模型行为不确定。阅读 Agent 源码时，不能像读普通脚本一样从头到尾逐行看，而应该先建立运行链路，再沿着任务生命周期拆解关键机制。

一套有效的源码阅读方法应该包括：先看产品入口，再画任务生命周期，然后追踪 Agent Loop，接着拆解工具系统、上下文构造、状态与记忆、安全边界、可观测性和前端交互。读完后，要把源码转化成自己的架构素材：运行链路图、模块职责表和设计取舍总结。

源码研究的价值不在于“看过多少项目”，而在于能否从项目中提炼可迁移的机制。一个优秀的 Agent 开发者，不是简单复制别人的实现，而是能判断哪些设计适合自己的场景，哪些设计需要改造，哪些设计暂时不应该引入。

从下一章开始，我们会把这套阅读方法应用到 OpenClaw、Hermes、Claude Code、Codex 这类代表性 Agent 形态中，分析它们各自的学习价值，并说明如何从成熟项目中提炼自己的 Agent Runtime。
