# 第 20 章：从 Demo 到产品：Agent 系统如何持续演进

到这里，全书已经走过了完整路径。我们从“为什么要系统学习 Agent”开始，区分了 Chatbot、Tool Calling 与 Agent；然后拆解了 Agent 的能力模型、适用边界、Agent Loop、工具系统、上下文工程、规划、记忆、RAG、长期任务、人机协同、评估和可观测性；接着学习了如何阅读 Agent 项目源码，如何从成熟项目中提炼自己的 Agent Runtime；最后用研究 Agent、外贸客户开发 Agent 和代码开发 Agent Mini 版完成了三个综合项目。

如果说前面章节解决的是“Agent 系统如何构建”，那么本章要解决的是另一个更现实的问题：

> 一个 Agent Demo 如何演进成真正有人使用、能持续迭代、能产生价值的产品？

这一步往往比写出第一个 Demo 更难。

很多 Agent 项目都停在 Demo 阶段。它们在演示中看起来很聪明，可以自动搜索、自动规划、自动写文件、自动调用工具。但一旦进入真实用户、真实数据、真实权限、真实成本和真实失败场景，问题就会集中暴露：结果不稳定、执行不可控、用户不知道它在做什么、失败无法复盘、成本无法控制、权限风险过高、业务闭环不完整。

所以，本章的核心不是再增加一个新模块，而是把全书内容收束成一套产品演进方法。

我们要回答这些问题：

```text
1. Agent Demo 和 Agent 产品的本质区别是什么？
2. 为什么很多 Agent 项目看起来先进，却无法长期使用？
3. 从 CLI Demo 到 Web 产品，需要补哪些能力？
4. 从单用户到多用户，需要补哪些能力？
5. 从一次性任务到长期任务，需要补哪些能力？
6. 从模型能力到业务价值，需要如何设计闭环？
7. 一个个人开发者应该如何用 90 天构建 Agent 能力？
8. 一个真实项目应该如何用 6 个月逐步演进？
```

本章既是总结，也是起点。读完本章，你应该能把本书的方法迁移到自己的项目中：外贸 Agent、教育 Agent、代码 Agent、个人助理、机器人任务管理、企业流程自动化，或者你未来会遇到的任何 Agent 场景。

## 20.1 Demo 和产品的区别

Agent Demo 的目标是证明“这件事看起来可以做”。

Agent 产品的目标是证明“这件事可以稳定地为用户创造价值”。

这两者差别非常大。

Demo 可以容忍偶然成功。只要一次演示顺利，观众就会觉得惊艳。产品必须面对大量重复使用。用户不会因为它偶尔成功而持续付费，用户关心的是今天能不能用、明天能不能用、失败后能不能恢复、结果能不能信任。

Demo 可以忽略边界。比如一个浏览器 Agent 自动点击网页、填写表单、发送请求，看起来很强。但产品必须明确哪些网站可以访问，哪些动作需要确认，哪些数据不能外传，遇到验证码怎么办，登录凭证如何保存，操作记录如何审计。

Demo 可以忽略成本。一个任务调用几十次模型、跑十分钟、消耗大量 token，在演示中可以接受。产品必须关心单位任务成本、用户付费能力、并发量、延迟、失败重试和资源消耗。

Demo 可以忽略用户体验。开发者自己看日志就能理解 Agent 在干什么，但普通用户需要任务进度、结果列表、审批按钮、失败原因、历史记录和可修改配置。

Demo 可以忽略长期状态。一次性搜索客户很容易，持续半年维护客户线索、去重、跟进、记录回复和更新评分就难得多。

因此，从 Demo 到产品，至少要补齐六类能力：

```text
1. 稳定性：任务能否重复成功？
2. 可控性：高风险动作是否受限？
3. 可解释性：用户是否知道 Agent 为什么这样做？
4. 可恢复性：失败后能否继续、回滚或复盘？
5. 可运营性：成本、质量、反馈、日志是否可管理？
6. 业务闭环：结果是否真的进入用户工作流程？
```

一个 Agent Demo 可以只展示智能，一个 Agent 产品必须提供信任。

## 20.2 Agent 产品的第一原则：不要卖“自主”，要卖“结果”

很多 Agent 项目的宣传喜欢强调“自主”。例如：自主搜索、自主规划、自主执行、自主学习。

自主性当然重要，但用户真正购买的不是自主性，而是结果。

外贸业务员不关心 Agent 是否用了多复杂的规划算法，他关心的是：有没有找到更好的客户？客户是否真实？联系方式是否可用？开发信是否能提高回复率？是否减少了重复劳动？是否降低了漏跟进？

老师不关心教育 Agent 是否使用了多智能的推理模型，他关心的是：学生薄弱点是否更清楚？作业批改是否更快？错题是否能转化为复习计划？家长沟通是否更省事？学生成绩和学习习惯是否改善？

开发者不关心代码 Agent 是否看起来像真人程序员，他关心的是：是否减少了查文件时间？是否能安全修改代码？测试是否通过？diff 是否清晰？是否不会乱改项目？

所以，Agent 产品的第一原则是：

> 不要把“Agent 很自主”当成价值主张，要把“它帮用户完成了什么工作”当成价值主张。

这会影响产品设计。

如果你做外贸 Agent，首页不应该只写“自主 AI Agent 平台”，而应该写：

```text
每天为你筛选 20 个高匹配海外潜在客户，生成可审核开发信，并自动提醒跟进。
```

如果你做教育 Agent，不应该只写“智能学习 Agent”，而应该写：

```text
帮助老师每天 10 分钟掌握学生错因、薄弱知识点和待干预学生名单。
```

如果你做代码 Agent，不应该只写“自主编程智能体”，而应该写：

```text
帮你读取项目、生成修改计划、完成小范围代码修改，并通过测试后交付 diff。
```

价值主张越具体，Agent 的设计边界越清晰。边界越清晰，产品越容易落地。

## 20.3 从 CLI 到 Web UI：不是换皮，而是交互模型变化

很多 Agent 项目最早都是 CLI 形态。CLI 很适合开发阶段，因为它简单、直接、便于调试。用户输入目标，终端显示 Agent 执行日志，开发者可以快速观察工具调用和错误。

但 CLI 并不适合所有用户，也不适合所有任务。

当 Agent 进入真实业务场景，用户需要的不只是输入命令，而是持续管理任务。例如外贸客户开发 Agent 需要展示客户列表、评分、来源、审批状态、触达历史、跟进提醒。教育 Agent 需要展示学生画像、错题分布、任务完成情况和老师干预记录。代码 Agent 需要展示 diff、测试结果、文件变更和回滚选项。

这意味着 Web UI 不是 CLI 的简单图形化，而是 Agent 工作流的产品化表达。

一个 Agent Web UI 通常需要包含以下区域：

```text
1. 目标输入区：用户定义任务目标和约束。
2. 任务进度区：展示当前执行阶段。
3. 中间结果区：展示候选结果、搜索结果、分析结果。
4. 审批队列区：展示需要用户确认的动作。
5. 历史记录区：查看过去任务和结果。
6. 设置区：管理工具权限、偏好、数据源和通知。
7. 反馈区：用户对结果打分、纠错、补充信息。
```

以外贸 Agent 为例，Web UI 不应该只是一个聊天框。它至少需要：

```text
- 客户线索表格；
- 客户详情页；
- 商机评分解释；
- 邮件草稿审批；
- 跟进时间线；
- 黑名单和已联系名单；
- 目标国家和产品设置；
- 每日新增客户报告。
```

聊天框仍然有价值，但它更适合做自然语言控制平面。例如用户可以说：

```text
把目标市场从阿联酋改成沙特和波兰。
以后优先找批发商，不要找零售店。
这批客户里评分低于 60 的先隐藏。
把开发信语气改得更简洁。
```

结构化界面负责稳定展示和操作，自然语言界面负责灵活控制和解释。两者不是替代关系，而是并行关系。

这是 Agent 产品设计的一个重要方向：

> 聊天不是全部界面，聊天是控制平面；结构化界面才是业务工作台。

## 20.4 从单次任务到长期任务

一次性 Agent 任务相对简单。用户输入目标，Agent 执行，返回结果，任务结束。

长期 Agent 难得多。它要持续运行、记住历史、避免重复、响应变化、处理失败、定期总结、等待人工审批。

以外贸 Agent 为例，一次性任务是：

```text
帮我找 30 个沙特五金客户。
```

长期任务是：

```text
每天帮我在沙特、阿联酋和波兰搜索新的五金批发商，排除已联系客户，优先评分高的客户，生成开发信草稿，每天下午 5 点给我审批，三天后自动提醒跟进未回复客户。
```

长期任务需要新增很多能力：

```text
- 任务调度；
- 状态持久化；
- 历史去重；
- 结果增量更新；
- 失败重试；
- 通知系统；
- 人工审批；
- 记忆更新；
- 周期性报告；
- 暂停、恢复和取消。
```

如果没有这些能力，Agent 就只是一个一次性工具，而不是一个长期工作者。

长期任务还有一个容易忽略的问题：用户的目标会变化。

例如，外贸 Agent 最初被要求找沙特客户。几天后用户发现沙特回复率不高，希望转向波兰。又过几天，用户发现批发商质量不如工程用品分销商，于是调整客户类型。Agent 必须支持目标变更，并保留变更历史。

这就要求系统区分三类配置：

```text
1. 长期偏好：例如开发信语气简洁、优先找批发商；
2. 当前任务配置：例如本周目标国家是沙特和波兰；
3. 临时指令：例如今天先不要发送提醒。
```

把三者混在一起，会导致记忆污染。临时指令不能被永久记住，长期偏好也不能轻易被一次任务覆盖。

因此，从单次任务走向长期任务，本质上是从“模型执行”走向“任务系统”。

## 20.5 从单用户到多用户

很多个人项目只考虑单用户。这在学习阶段完全合理。但如果要变成产品，多用户问题迟早会出现。

多用户不是简单地给数据库加一个 `user_id`。它会影响权限、数据隔离、计费、团队协作、审计、配置和安全。

例如，一个外贸公司可能有老板、业务主管、业务员和运营助理。

老板希望看总体客户开发进展；业务主管希望分配目标市场；业务员希望审批自己的邮件草稿；运营助理可能只负责整理线索，不能发送邮件。

这时系统需要角色权限：

```text
老板：查看所有数据，管理配置；
主管：分配任务，审批高风险动作；
业务员：查看和处理自己的客户；
助理：整理线索，不能触达客户。
```

代码 Agent 也一样。个人开发者可以允许 Agent 修改本地文件，但团队项目需要更严格权限。初级开发者可能只能让 Agent 生成 patch，不能直接合并；核心仓库必须通过 PR；生产配置文件不能修改；安全相关代码需要高级审批。

多用户还意味着数据隔离。一个用户的客户名单不能被另一个用户看到；一个团队的代码仓库上下文不能泄漏到另一个团队；一个学生的学习记录不能被无关老师访问。

因此，多用户 Agent 产品至少需要：

```text
- 用户身份；
- 团队空间；
- 角色权限；
- 数据隔离；
- 操作审计；
- 配额与计费；
- 工具权限配置；
- 敏感数据管理。
```

这些内容看起来不像 Agent 技术，但它们决定了 Agent 是否能真正进入企业和团队场景。

## 20.6 从模型能力到业务闭环

很多 Agent 项目的另一个问题，是只展示模型能力，没有进入业务闭环。

业务闭环是指：Agent 的输出能够进入用户真实工作流，并对结果产生持续影响。

以研究 Agent 为例，如果它只是生成报告，用户看完就结束，那么闭环较弱。如果它能把报告中的行动项转成任务、把不确定点加入后续调研、把用户反馈写回评估集，那么闭环更强。

以外贸 Agent 为例，真正的闭环不是“生成客户表格”，而是：

```text
发现客户 → 筛选评分 → 生成触达建议 → 人工审批 → 发送记录 → 回复理解 → 跟进计划 → 成交反馈 → 更新评分策略
```

如果没有回复理解和成交反馈，Agent 就不知道哪些客户真的有价值，也无法改进筛选策略。

以教育 Agent 为例，闭环不是“讲解错题”，而是：

```text
做题 → 批改 → 错因分析 → 学生画像更新 → 推荐练习 → 老师干预 → 再测试 → 效果评估
```

如果没有再测试和效果评估，系统就无法证明学习真的改善。

以代码 Agent 为例，闭环不是“生成代码”，而是：

```text
需求 → 计划 → 修改 → 测试 → review → 合并 → 线上反馈 → 经验沉淀
```

业务闭环决定了 Agent 的价值能否积累。没有闭环，Agent 每次都像重新开始。有闭环，Agent 才能越用越贴近业务。

这也是为什么本书反复强调 Memory、Evaluation 和 Observability。它们不是高级装饰，而是业务闭环的基础。

## 20.7 Agent 产品的核心指标

Agent 产品不能只看模型输出质量。它需要一组围绕任务价值的指标。

可以分成五类。

第一类是任务效果指标。

```text
- 任务完成率；
- 结果准确率；
- 结果可用率；
- 人工采纳率；
- 关键业务转化率。
```

外贸 Agent 可以看高质量客户比例、邮件审批通过率、回复率、有效询盘数。教育 Agent 可以看学生任务完成率、错题复发率、薄弱点改善率。代码 Agent 可以看测试通过率、diff 采纳率、回滚率。

第二类是效率指标。

```text
- 单任务耗时；
- 人工节省时间；
- 平均审批时间；
- 重复劳动减少量；
- 单位成本。
```

Agent 的价值通常不是完全替代人，而是提升人的处理效率。效率指标非常重要。

第三类是安全指标。

```text
- 高风险动作审批率；
- 未授权工具调用次数；
- 敏感数据访问次数；
- 错误发送、错误修改、错误删除事件；
- prompt injection 触发与拦截情况。
```

第四类是可靠性指标。

```text
- 任务失败率；
- 平均重试次数；
- 超时率；
- 恢复成功率；
- 用户中断次数。
```

第五类是用户信任指标。

```text
- 用户是否查看执行过程；
- 用户是否修改 Agent 结果；
- 用户是否愿意授权更多工具；
- 用户是否持续使用；
- 用户是否把 Agent 输出用于真实工作。
```

一个 Agent 产品早期最应该关注的不是“看起来多智能”，而是“用户是否愿意把真实任务交给它”。

## 20.8 Agent Runtime 的演进路线

在第 16 章，我们设计了自己的 Agent Runtime。到产品阶段，这个 Runtime 需要持续演进。

第一阶段是单任务 Runtime。

它只需要支持一次任务执行：接收目标、调用模型、使用工具、返回结果。适合学习和 Demo。

第二阶段是可观察 Runtime。

它增加 trace、日志、工具调用记录、成本统计和失败原因。适合调试和内部试用。

第三阶段是可控 Runtime。

它增加权限、审批、工具分级、dry-run、最大步数、超时和取消。适合处理真实工具。

第四阶段是长期 Runtime。

它增加任务队列、调度、checkpoint、resume、长期记忆、通知和历史结果。适合持续任务。

第五阶段是多用户 Runtime。

它增加用户、团队、角色、数据隔离、配额、审计和管理后台。适合产品化。

第六阶段是可学习 Runtime。

它把用户反馈、任务结果、失败案例、评估数据写回系统，形成经验沉淀。适合长期迭代。

可以总结为：

```text
能跑 → 可看 → 可控 → 可持续 → 可协作 → 可学习
```

这条路线比一开始追求“大而全”更稳。

## 20.9 个人开发者的 90 天学习路线

如果你是个人开发者，想真正掌握 Agent 构建能力，可以按 90 天推进。

### 第 1–15 天：建立基础认知

目标不是写复杂代码，而是建立判断力。

你需要完成：

```text
- 阅读本书第 1–3 章；
- 区分 Chatbot、Tool Calling 和 Agent；
- 分析 20 个任务是否适合 Agent；
- 画出一个自己想做的 Agent 的能力模型；
- 写一页纸产品合同。
```

阶段产出：

```text
agent-idea-one-page.md
agent-task-boundary.md
agent-capability-map.md
```

### 第 16–35 天：手写核心机制

目标是理解 Agent Runtime。

你需要完成：

```text
- 实现最小 Agent Loop；
- 实现 Tool Registry；
- 实现 Context Manager；
- 实现 ReAct 与 Planner-Executor 两种模式；
- 实现基础执行日志。
```

阶段产出：

```text
mini-agent-runtime-v1/
```

### 第 36–55 天：加入长期能力

目标是让 Agent 从一次性任务变成可持续任务。

你需要完成：

```text
- 实现 Memory Store；
- 实现简单 RAG；
- 实现任务状态机；
- 实现 checkpoint；
- 实现简单调度。
```

阶段产出：

```text
mini-agent-runtime-v2/
```

### 第 56–70 天：加入安全与评估

目标是让 Agent 更接近真实系统。

你需要完成：

```text
- 工具权限分级；
- 审批队列；
- 最大步数和超时；
- eval harness；
- trace 和 replay。
```

阶段产出：

```text
mini-agent-runtime-v3/
```

### 第 71–90 天：完成一个真实项目

选择一个项目，不要同时做多个。

推荐三选一：

```text
1. 外贸客户开发 Agent；
2. 代码开发 Agent Mini 版；
3. 教育监督闭环 Agent。
```

## 20.10 一个真实项目的 6 个月演进路线

如果你希望把 Agent 做成真实产品，可以按 6 个月规划。

### 第 1 个月：定义场景和最小价值闭环

不要先做平台，先做一个高频、明确、有价值的场景。

例如外贸 Agent 的最小价值闭环可以是：

```text
输入产品和国家 → 找客户 → 评分 → 生成开发信 → 人工审批 → 记录触达
```

这个阶段最重要的是验证：用户是否真的需要这个结果？是否愿意每天使用？结果是否能进入真实工作流？

### 第 2 个月：实现可控 Agent Runtime

实现核心运行时：

```text
- 任务状态；
- 工具系统；
- 上下文管理；
- 审批队列；
- 日志；
- 失败处理。
```

这一阶段不要过度追求 UI 精美，重点是让系统可控。

### 第 3 个月：打磨业务数据结构

Agent 产品的核心不是模型，而是业务对象。

外贸 Agent 的业务对象包括：

```text
Lead
CompanyProfile
Contact
OpportunityScore
EmailDraft
OutreachRecord
FollowUpTask
```

教育 Agent 的业务对象包括：

```text
StudentProfile
MistakeRecord
KnowledgePoint
LearningTask
ReviewPlan
TeacherIntervention
```

代码 Agent 的业务对象包括：

```text
RepoContext
ChangePlan
FilePatch
TestResult
ReviewDecision
Checkpoint
```

业务对象清晰，Agent 输出才能结构化、可保存、可评估。

### 第 4 个月：加入长期任务和反馈闭环

实现调度、历史、记忆和反馈。

```text
- 每日任务；
- 去重；
- 历史记录；
- 用户反馈；
- 策略调整；
- 周报。
```

这一步决定产品是否能持续使用。

### 第 5 个月：做评估和质量提升

建立评估集和质量指标。

```text
- 任务成功率；
- 结果采纳率；
- 错误类型统计；
- 成本统计；
- 用户反馈分析；
- A/B 测试不同策略。
```

不要只凭感觉优化 Agent。没有评估，系统会越改越乱。

### 第 6 个月：产品化与商业化准备

补齐产品能力：

```text
- 用户系统；
- 权限；
- 团队空间；
- 计费；
- 数据导入导出；
- 管理后台；
- 隐私与合规说明；
- 客户 onboarding 流程。
```

到第 6 个月，你应该已经知道这个产品是否值得继续投入。如果用户愿意持续使用，并且真实节省时间或带来收入，就可以继续扩展。

## 20.11 如何选择自己的第一个 Agent 项目

很多人学习 Agent 后，会想做一个“大而全 AI 员工”。这个方向有吸引力，但不适合作为第一个落地项目。

第一个 Agent 项目应该满足几个条件。

第一，任务足够具体。比如“为钢卷尺找海外批发商”比“帮我做外贸”更适合。

第二，结果可以验证。比如客户是否真实、测试是否通过、错题是否减少，都是相对可评估的。

第三，工具风险可控。不要一上来就让 Agent 直接付款、删库、发合同。

第四，用户反馈周期短。最好当天或一周内能知道结果好不好。

第五，你自己懂业务。Agent 不是凭空创造业务价值，它需要嵌入真实工作流。如果你不懂业务，就很难判断输出是否有用。

根据这些标准，适合个人开发者的项目包括：

```text
- 外贸客户线索筛选 Agent；
- 代码仓库小任务修改 Agent；
- 学生错题归因和复习计划 Agent；
- 行业信息监控和报告 Agent；
- 电商选品数据整理 Agent；
- 企业内部文档问答与任务生成 Agent。
```

不适合作为第一个项目的包括：

```text
- 完全通用的个人助理；
- 自动经营一家公司；
- 完全替代程序员；
- 全自动销售闭环；
- 无边界浏览器操作 Agent；
- 高风险金融或医疗决策 Agent。
```

第一项目越具体，越容易成功。成功后再扩展成平台。

## 20.12 本书方法如何迁移到 Athena

如果你正在构建一个 AI 员工系统，例如 Athena，那么本书的内容可以直接迁移。

Athena 这类系统的核心不是一个聊天框，而是一套角色化、长期化、可控化的 Agent 工作系统。

可以这样映射：

```text
Agent 能力模型 → Athena 员工能力边界
Agent Loop → Athena 单次任务执行机制
Tool Registry → Athena 技能和工具权限
Context Manager → Athena 工作上下文
Memory Store → Athena 员工长期记忆
Scheduler → Athena 主动工作和周期任务
Approval System → Athena 高风险动作审批
Eval Harness → Athena 工作质量评估
Observability → Athena 工作日志和复盘
Dashboard → Athena 管理工作台
```

一个 AI 员工不能只靠“人格设定”。它必须有职责、目标、工具、记忆、权限、评估和复盘。

例如，一个外贸员 Athena 可以有：

```text
职责：持续发现潜在客户并维护触达节奏。
工具：搜索、网页读取、客户数据库、邮件草稿、CRM 写入。
权限：可生成邮件草稿，不可自动发送；可新增客户，不可删除客户。
记忆：产品信息、客户历史、用户偏好、成功话术。
评估：客户质量、回复率、审批通过率、有效询盘。
复盘：每周总结搜索策略和客户反馈。
```

这就比“你是一个外贸专家，请帮我找客户”强得多。

本书最终要训练的，就是这种系统设计能力。

## 20.14 写在最后：Agent 是新的软件形态，但仍然是软件工程

Agent 给软件带来了新的可能性。它让系统能够理解自然语言目标，处理不确定信息，调用工具，持续执行，并与人类协作。这确实是重要变化。

但 Agent 并没有让软件工程失效。恰恰相反，Agent 越强，越需要工程约束。

没有状态管理，Agent 会混乱。

没有工具权限，Agent 会危险。

没有上下文工程，Agent 会遗忘或误解。

没有记忆治理，Agent 会污染长期行为。

没有审批，Agent 会把错误变成真实损失。

没有评估，Agent 无法持续改进。

没有产品界面，用户无法真正使用。

所以，本书反复强调：

> Agent 不是魔法，而是一种新的软件系统形态。

它的核心不是把一切交给模型，而是把模型能力放进一个可控、可观察、可评估、可演进的系统中。

真正掌握 Agent 的人，不是只会写 prompt 的人，也不是只会调用框架的人，而是能把任务、模型、工具、记忆、权限、评估和业务闭环组合起来的人。

这也是本书希望帮助你建立的能力。

## 本章总结

从 Demo 到产品，是 Agent 项目最关键也最容易被低估的一步。Demo 证明的是可能性，产品证明的是稳定价值。一个 Agent 产品不仅要能执行任务，还要可控、可解释、可恢复、可评估、可运营，并且真正进入用户的业务闭环。

本章把全书内容收束为一条演进路线：从能跑，到可看，到可控，到可持续，到可协作，再到可学习。这个路线适用于研究 Agent、外贸 Agent、教育 Agent、代码 Agent，也适用于更大的 AI 员工系统。

Agent 的未来不会只是一个更强的聊天框，而会是一种新的软件系统形态。它把自然语言目标、模型推理、工具执行、长期记忆、人机协同和业务流程结合起来。真正有价值的 Agent，不是无限自主的黑箱，而是在明确边界内持续创造结果的工作系统。

这本书到这里完成了第一轮闭环。你已经从概念、机制、工程、源码、项目和产品化六个层面学习了 Agent 系统。接下来真正重要的，不是继续看更多概念，而是选择一个具体场景，做出一个能跑、可控、可评估、能被真实用户使用的小系统。

最好的学习方式，始终是构建。
