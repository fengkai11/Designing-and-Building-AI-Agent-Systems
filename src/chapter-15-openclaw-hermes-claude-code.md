前一章我们建立了一套阅读 Agent 项目源码的方法：先看产品入口，再画任务生命周期，然后追踪 Agent Loop、工具系统、上下文构造、状态与记忆、安全边界、可观测性和前端交互。本章要进一步回答一个更具体的问题：面对成熟 Agent 产品和开源项目，我们到底应该学什么？

很多人研究 Agent 时，会列出一批名字：OpenClaw、Hermes、Claude Code、Codex、LangGraph、AutoGPT、CrewAI、Devin、OpenHands 等。名字很多，方向也不同。如果没有分类，很容易被表象带跑：看到 OpenClaw 支持很多渠道，就以为重点是“接入平台”；看到 Claude Code 能改代码，就以为重点是“模型写代码”；看到 Codex 可以并行处理任务，就以为重点是“多 Agent”；看到某些开源框架有复杂图结构，就以为重点是“流程编排”。

事实上，每一个成熟项目都只是 Agent 系统空间中的一个切片。我们研究它们，不是为了复制一个一模一样的产品，而是为了从不同切片中提炼可迁移能力。

本章会重点讨论四类代表：

1. OpenClaw：个人助理型、长期运行型、多渠道接入型 Agent；
2. Hermes：在本书中作为“消息/助理型 Agent”的代表性形态讨论，重点学习任务入口、角色身份和人机通信模式；
3. Claude Code：代码开发型、终端/IDE 协作型 Agent；
4. Codex：代码任务委派型、本地 CLI 与云端并行任务型 Agent。

需要提前说明：本章不会编造闭源产品的内部实现。对于公开文档和开源仓库能够确认的内容，我们会明确作为事实使用；对于无法确认的内部机制，只从外部行为和 Agent 工程原则进行推断，并标注为“可学习模式”而不是“源码事实”。这是技术写作中非常重要的边界。

## 15.1 学成熟项目，不是学表面功能，而是学系统取舍

一个成熟 Agent 项目通常会给人很强的冲击感。它可能支持多平台消息，能读写代码，能操作浏览器，能长期运行，能连接外部工具，能在后台处理任务。初学者看到这些功能后，最容易产生的想法是：我也要全部做一遍。

这是一个危险想法。

真实项目的功能，是在特定目标、用户、风险和工程历史下形成的。脱离背景复制功能，很容易把自己的系统做复杂。

例如，OpenClaw 类个人助理如果强调“在你已有渠道中回答你”，那么多渠道接入就是核心产品价值。用户可能通过 Telegram、Slack、iMessage、微信或其他消息渠道与助手交互。系统需要把不同渠道消息统一成内部事件，再把 Agent 回复路由回对应渠道。这对个人助理非常重要。

但如果你要做的是外贸客户开发 Agent，第一阶段未必需要接入十几个聊天渠道。你更需要的是客户搜索、线索去重、商机评分、开发信审批、触达历史和跟进调度。此时盲目复制多渠道接入，可能会浪费精力。

再比如，Claude Code 的核心场景是代码开发。它需要理解代码库、编辑文件、运行命令、处理测试失败、展示 diff、集成 IDE 或终端。这些能力对代码 Agent 极其关键。但如果你做教育 Agent，就不需要 shell 工具和 git diff。你更需要的是题目结构化、错因分类、学生画像、教师审批和学习计划。

所以，研究成熟项目时必须问：

```text
它服务的核心场景是什么？
它的用户是谁？
它解决了什么高频问题？
它为什么需要这些模块？
哪些模块对我的项目也成立？
哪些模块只是它场景下的特殊需求？
```

学项目，不是学“它有什么”，而是学“它为什么这样设计”。

## 15.1.1 事实、推断与架构启发的边界

本章讨论成熟项目时，必须先划清三条线：

第一条线是公开事实。公开事实来自项目 README、官方文档、官方博客、公开产品页面或可直接验证的产品行为。例如，OpenClaw README 明确把 OpenClaw 描述为运行在自己设备上的 personal AI assistant，并说明 Gateway 是控制平面，产品是 assistant；Claude Code 官方文档明确说它是可以读取代码库、编辑文件、运行命令并集成开发工具的 agentic coding tool；OpenAI Codex 文档明确说明 Codex 可以读、改、运行代码，Codex cloud 可以在云端后台并行处理任务，Codex CLI 可以在本地终端选定目录中读写和运行代码。

第二条线是合理推断。合理推断不是事实陈述，而是根据公开行为和常见工程模式做出的架构分析。比如，一个代码 Agent 能稳定修改仓库，通常需要 repo context、文件读写工具、shell 权限控制、diff 展示、测试反馈和人工确认；但除非官方文档明确披露，我们不能声称它内部一定采用某个具体类名、队列实现或状态机结构。

第三条线是架构启发。架构启发是本书从成熟产品中提炼出来、用于设计自己 Runtime 的模式。比如“控制平面与执行平面分离”“自然语言入口 + 结构化工作台”“后台任务 + 前台审批”“工具风险分级”。这些模式可以借鉴，但不等于被分析对象一定逐字实现了同名模块。

| 对象 | 可确认公开事实 | 本书中的合理推断 | 仅作为架构启发的内容 |
|---|---|---|---|
| OpenClaw | 公开 README 将其描述为运行在自己设备上的个人 AI 助理，支持多渠道、语音、live Canvas，并强调 Gateway 是 control plane，product 是 assistant。 | 多渠道入口需要统一消息路由、任务分发和状态保存；live Canvas 暗示前端不只是聊天框，而是可承载交互式工作面。 | 控制平面 / 执行平面分离；消息入口与工作台并行；长期任务需要状态与记忆。 |
| Hermes 类助理 | 本书未绑定唯一可核验的 Hermes 开源仓库或官方产品。 | 只把它作为“消息型、角色型、助理型 Agent”的抽象形态来分析。 | 角色身份、职责边界、消息入口、长期上下文、通知与提醒。 |
| Claude Code | 官方文档将其描述为 agentic coding tool，可读取代码库、编辑文件、运行命令，并集成终端、IDE、桌面 app 和浏览器。 | 稳定代码修改需要 repo context、文件工具、shell 工具、测试反馈、diff 与人工 review。 | 代码 Agent 的工具权限分级、测试闭环、修改前计划、修改后 review。 |
| Codex | OpenAI 文档说明 Codex 可读、改、运行代码；Codex cloud 可在云端后台并行执行任务；Codex CLI 可在本地终端运行并操作选定目录。 | 云端任务模式需要沙箱、任务隔离、日志、测试输出和 diff 回传；本地 CLI 模式需要目录权限和用户确认机制。 | 任务委派、云端沙箱、并行任务、CLI / Web / IDE 多入口协同。 |

因此，本章不是要“揭秘”这些项目的内部实现，而是训练一种能力：把公开事实、工程推断和架构启发分开，并从中提炼自己的设计方法。

## 15.2 OpenClaw 的学习价值：个人助理、控制平面与长期运行

公开仓库对 OpenClaw 的定位非常清晰：它是运行在自己设备上的个人 AI 助理，可以在用户已经使用的渠道中响应，支持语音、移动端和 live Canvas；其 README 中还强调 Gateway 只是控制平面，产品本身是 assistant。

这几个关键词非常值得学习：

```text
personal assistant
run on your own devices
channels you already use
speak and listen
live Canvas
Gateway as control plane
assistant as product
```

它给我们的第一点启发是：Agent 的入口可以不是一个新 App，而是用户已有的通信渠道。

很多 AI 产品默认要用户打开一个新网页、新工作台或新 App。但个人助理类 Agent 的价值，恰恰可能来自“进入用户已有工作流”。如果用户平时就在 Telegram、Slack、微信、iMessage 或其他消息工具里处理事务，那么 Agent 出现在这些渠道中，会比单独打开一个后台更自然。

这对外贸 Agent 也有启发。外贸业务员可能不愿意每天打开一个复杂系统，但如果 Agent 能在企业微信、飞书、Slack 或邮件中推送“今日新增高价值客户”“待审批开发信”“某客户回复需跟进”，使用阻力会低很多。

第二点启发是：Gateway 与 Assistant 的分离。

“Gateway 是控制平面，产品是 assistant”这句话很关键。它说明系统可以把渠道接入、路由、认证、消息传输等能力作为控制平面，而真正的价值在 Agent 的任务执行能力上。

在自己的系统中，也可以采用类似思路：

```text
控制平面：用户、权限、任务入口、渠道、配置、审批、监控。
执行平面：Agent Runtime、工具调用、模型推理、任务状态、记忆。
产品价值：某个具体角色完成某类工作。
```

例如 Athena AI 员工系统中，控制平面可能是 Monitor 给员工设目标、看状态、审批动作；执行平面是员工 Agent 调用工具完成任务；真正的产品价值不是控制台本身，而是“AI 员工能持续完成某类工作”。

第三点启发是：多渠道并不等于多 Agent。

支持很多渠道，核心是消息统一和身份映射。系统需要把不同渠道的消息转换成统一内部格式，例如：

```json
{
  "channel": "telegram",
  "user_id": "u_123",
  "conversation_id": "c_456",
  "message_type": "text",
  "content": "帮我看一下今天有哪些待处理事项",
  "timestamp": "..."
}
```

进入内部系统后，Agent 不应该关心消息来自 Telegram 还是 Slack。它只处理标准化事件。回复时再由 channel adapter 转回具体平台。

这是一种非常通用的架构模式：

```text
Channel Adapter -> Normalized Event -> Agent Runtime -> Response Event -> Channel Adapter
```

第四点启发是：个人助理天然需要长期记忆和偏好系统。

一个个人助理如果每次都像第一次认识用户，就很难真正有价值。它需要知道用户的偏好、常用联系人、日程习惯、消息处理风格、哪些事情可以自动做、哪些事情必须确认。

但这也带来风险：个人助理接触的信息高度私人化，必须有清晰的隐私边界、权限控制和本地运行/自托管策略。OpenClaw “run on your own devices”的定位，正好说明个人助理类 Agent 对本地性和数据控制有天然需求。

这对企业 Agent 也有启发。外贸公司可能不愿意把客户名单、报价、邮件历史完全交给第三方云服务。教育机构也不愿意学生数据外泄。因此，自托管、本地存储、权限隔离会成为很多 Agent 产品的重要卖点。

第五点启发是：live Canvas 代表一种“可视化协作面”。

聊天消息适合短指令和短反馈，但不适合展示复杂状态。比如客户列表、任务流程、审批队列、代码 diff、学习报告，都需要结构化界面。Canvas 或工作台提供了一个人类可以查看、编辑、确认、接管的界面。

这说明成熟 Agent 产品往往不是纯聊天，也不是纯后台自动化，而是“自然语言入口 + 结构化工作台”的组合。

这和我们之前在 Athena 项目中讨论的“自然语言控制平面 + 结构化控制台并行”是一致的。

## 15.3 OpenClaw 对 Athena 和外贸 Agent 的具体启发

如果把 OpenClaw 的设计模式迁移到 Athena 或外贸 Agent，可以提炼出几条原则。

第一，Agent 不一定要从 Web 后台开始。

很多业务任务可以从消息入口触发。例如：

```text
“今天帮我找 20 个阿联酋五金批发商。”
“把昨天找到的客户按优先级发我。”
“这 5 封开发信先不要发，语气改得更短一点。”
“把已经拒绝的客户加入黑名单。”
```

这些指令通过聊天入口非常自然。但最终结果可能需要在表格或工作台里查看。

因此外贸 Agent 可以采用：

```text
聊天入口：发任务、改方向、问状态。
结构化工作台：看客户、审邮件、看历史、调配置。
后台 runtime：执行搜索、评分、跟进、记忆更新。
```

第二，多渠道接入应该后置，但统一事件模型应该前置。

第一版不一定支持很多平台，但内部最好从一开始就把输入抽象成事件。这样未来接入微信、飞书、邮件、Slack 时，不需要重写 Agent Runtime。

第三，控制平面和执行平面要分开。

Monitor 或用户配置目标、权限、审批策略；Agent Runtime 负责执行；业务界面负责展示结果。不要把所有逻辑堆在聊天框里。

第四，长期个人化能力要谨慎设计。

外贸 Agent 可以记住用户的产品、目标国家、客户偏好、邮件风格、黑名单、已触达客户。但不能随意记住临时信息，也不能把错误判断永久化。记忆写入需要规则和可编辑界面。

第五，always-on 不等于无限自主。

长期运行的 Agent 应该可以定时发现机会、推送待办、提醒跟进，但高风险动作仍要审批。比如自动搜索客户可以，自动群发邮件不应该默认允许。

## 15.4 Hermes 类助理的学习价值：消息型助理与角色身份

“Hermes”这个名字在不同上下文中可能指不同项目，也可能只是某个团队内部或社区中的 Agent 助理名称。因此，在没有指定唯一公开仓库和版本之前，本书不把 Hermes 当成某个可验证源码项目来讲内部实现，而把它作为一类“消息型助理 / 个人助理 Agent”的代表形态来分析。

Hermes 在希腊神话中是众神信使，这个命名本身就很适合助理型 Agent：它连接人和信息，连接任务和工具，连接外部世界和用户意图。

从 Agent 产品角度，Hermes 类系统的学习价值主要有四点。

第一，角色身份非常重要。

一个通用聊天机器人可以回答各种问题，但一个 Agent 需要有明确职责。它到底是秘书、代码助理、外贸业务员、教育助教、运营助手，还是项目经理？角色不同，权限、工具、记忆、评估标准都不同。

例如，一个“外贸业务助理 Hermes”应该具备：

```text
产品知识；
目标客户识别能力；
邮件草稿能力；
客户跟进记忆；
CRM 工具；
审批边界；
商机评分标准。
```

一个“代码助理 Hermes”则应该具备：

```text
repo 读取；
文件编辑；
测试运行；
diff 展示；
命令权限控制；
代码审查标准。
```

如果角色定义模糊，Agent 就会变成“什么都能聊，但什么都做不深”的助手。

第二，消息型助理需要短反馈和长任务结合。

用户在聊天中发指令，期待快速回应。但任务本身可能很长。一个好的消息型 Agent 不能让用户长时间等待，也不能只说“我在处理”。它应该快速返回任务理解和初始计划，然后在后台执行，并在关键节点推送状态。

例如：

```text
用户：帮我找 30 个沙特五金批发商。
Agent：收到。我会按“批发商/进口商/工程用品分销商”三类搜索，排除零售店。预计输出客户表、评分和开发信草稿。找到第一批候选后会先给你预览。
```

这里的核心不是“礼貌回复”，而是任务契约。Agent 告诉用户它理解了什么、准备怎么做、什么时候需要用户参与。

第三，消息型助理需要强状态意识。

在聊天中，用户经常会用省略表达：

```text
“刚才那批客户只保留阿联酋的。”
“第二封邮件语气太硬，改软一点。”
“把这个客户标记为不要再联系。”
```

如果系统没有任务状态和对象引用，就不知道“那批客户”“第二封邮件”“这个客户”指什么。消息型 Agent 必须把聊天中的自然语言指令映射到结构化对象。

第四，消息型助理必须支持纠偏。

用户经常在中途改变目标。比如一开始说找“中东客户”，后来发现范围太大，改成“先只做沙特和阿联酋”；一开始想找进口商，后来改成“优先工程用品分销商”。Agent 不能把计划写死，而要允许用户中途修改状态。

这也是 Hermes 类助理值得学习的地方：它不是一次性问答，而是连续任务中的沟通界面。

## 15.5 Claude Code 的学习价值：代码库上下文、工具权限与开发闭环

Claude Code 官方文档将其描述为一种 agentic coding tool，能够读取代码库、编辑文件、运行命令，并与终端、IDE、桌面应用和浏览器等开发工具集成。这个定位非常典型：它不是单纯“生成代码”的聊天模型，而是嵌入开发环境的代码 Agent。

代码 Agent 是 Agent 学习中最重要的类型之一，因为它把 Agent 的核心问题暴露得非常清楚。

第一，代码 Agent 必须处理大型上下文。

一个真实代码库可能有几百到几千个文件。用户说“修复登录 bug”，Agent 不可能把整个 repo 全塞进 prompt。它必须决定先看哪些文件：README、路由、auth 模块、测试、错误日志、配置文件。它需要搜索、索引、摘要和逐步读取。

这就是 repo context 问题。

学习 Claude Code 类产品时，最值得关注的不是它能不能写代码，而是它如何选择上下文。一个好的代码 Agent 应该像开发者一样：先理解项目结构，再定位相关模块，再读具体文件，而不是盲目生成。

第二，代码 Agent 必须使用真实工具。

写代码不是只输出文本。它要读取文件、修改文件、运行测试、执行格式化、查看 git diff、可能还要安装依赖。每个工具都有副作用。

这使代码 Agent 成为学习工具权限的最佳场景。

例如：

```text
read_file：低风险，可以自动执行。
search_files：低风险，可以自动执行。
write_file：中风险，最好生成 diff。
run_tests：中风险，可以自动执行但要限制时间。
run_shell：高风险，需要命令限制和审批。
delete_file：高风险，必须确认。
git commit / push：高风险，必须确认。
```

第三，代码 Agent 必须形成开发闭环。

一个普通代码生成器输出一段代码就结束。代码 Agent 不应该这样。它应该形成闭环：

```text
理解需求 -> 读取代码 -> 制定计划 -> 修改文件 -> 运行测试 -> 修复失败 -> 展示 diff -> 用户 review
```

这个闭环非常值得迁移到其他业务 Agent。

外贸 Agent 也可以有类似闭环：

```text
理解目标 -> 搜索客户 -> 筛选去重 -> 评分 -> 生成邮件 -> 审批 -> 记录触达 -> 跟进反馈
```

教育 Agent 也可以有类似闭环：

```text
识别学习目标 -> 分析错题 -> 生成练习 -> 学生完成 -> 批改反馈 -> 更新画像 -> 老师干预
```

第四，代码 Agent 必须支持可审查输出。

代码修改不能只给最终结果，必须展示 diff、解释变更、说明测试情况。用户需要 review，而不是盲目信任。

这对任何 Agent 都成立。外贸 Agent 生成客户名单时，也应该展示来源和判断依据；教育 Agent 生成学习计划时，也应该展示依据和目标；运营 Agent 生成策略时，也应该展示数据来源。

第五，代码 Agent 的失败是可验证的。

很多业务 Agent 的结果质量难以自动判断，但代码 Agent 有测试、类型检查、lint、构建结果。这使它成为 Agent Evaluation 的好训练场。

如果测试失败，Agent 可以读取错误日志并继续修复。这种“执行—反馈—修正”的闭环，是 Agent 能力的重要体现。

## 15.6 Codex 的学习价值：任务委派、云端执行与并行开发

Codex 是另一个重要的代码 Agent 代表。OpenAI 的 Codex 文档将其描述为可以阅读、编辑和运行代码的 coding agent；Codex cloud 可以在自己的云环境中后台执行任务，并且支持并行处理多个任务。Codex CLI 则是运行在本地终端的轻量代码 Agent。

Codex 给我们的启发有几类。

第一，Agent 可以作为“任务委派对象”。

传统 AI 编程助手更像 pair programmer：用户在编辑器里边写边问。Codex cloud 这类形态更像把一个任务交给后台工程师：用户描述问题，Agent 在独立环境中处理，最后提交结果或 PR。

这对 Agent 产品形态非常重要。

很多业务 Agent 也可以从“聊天助手”升级为“任务委派系统”。用户不是问它一句话，而是交给它一个任务：

```text
今天帮我整理 50 个墨西哥潜在客户，按价值排序，生成可审批邮件草稿。
```

Agent 在后台执行，用户稍后 review。

第二，云端执行强调环境配置。

代码任务不是只靠模型，还需要运行环境。依赖能否安装？测试如何运行？是否有数据库？是否有环境变量？是否允许访问互联网？这些都会影响任务成败。

这对非代码 Agent 也有启发。外贸 Agent 的“执行环境”包括搜索工具、浏览器、代理、CRM、邮件草稿系统、产品知识库、黑名单。教育 Agent 的执行环境包括题库、学生数据、课程标准、错题库、老师工作台。

Agent 能力不仅取决于模型，也取决于环境配置。

第三，并行任务需要隔离。

Codex cloud 这类系统支持多个任务并行处理代码时，必须处理 worktree、分支、环境隔离和结果合并。否则不同任务会互相影响。

业务 Agent 同样需要隔离。例如外贸 Agent 同时执行“沙特客户开发”和“波兰客户开发”两个任务时，不能混淆国家、客户列表、邮件风格和触达历史。教育 Agent 同时处理多个学生，也必须隔离学生画像和学习记录。

第四，任务结果应该进入 review 流程。

代码 Agent 的输出往往不是直接上线，而是生成 diff、PR 或建议，由人类审查。这个模式特别值得所有 Agent 借鉴。Agent 可以完成大量初步工作，但最终高风险动作进入 review。

第五，本地 CLI 和云端 Agent 是两种互补形态。

本地 CLI 适合低延迟、贴近开发者当前环境、直接操作本地文件。云端 Agent 适合长任务、并行任务、后台执行和远程协作。自己的 Agent 产品也可以类似设计：

```text
本地/私有部署：处理敏感数据、文件、企业内部系统。
云端任务：处理可外包、可并行、低敏感的搜索和整理。
```

不要把 Agent 固定理解成一个界面。它可以是 CLI、Web、IDE、聊天入口、后台 worker，也可以同时存在多种形态。

## 15.7 四类 Agent 形态的对照

我们可以把 OpenClaw、Hermes 类助理、Claude Code、Codex 放在一个对照表中看。

```markdown
| 类型 | 主要入口 | 核心任务 | 关键机制 | 最值得学习的地方 |
|---|---|---|---|---|
| OpenClaw | 多聊天渠道 / 本地设备 / Canvas | 个人助理、消息处理、长期陪伴 | 渠道适配、控制平面、长期运行、本地性 | Agent 如何进入用户已有工作流 |
| Hermes 类助理 | 消息入口 / 角色化对话 | 任务沟通、状态跟进、个人或业务助理 | 角色身份、短反馈、长任务状态、纠偏 | Agent 如何通过消息完成任务协作 |
| Claude Code | 终端 / IDE / Web / 桌面 | 代码理解、修改、测试、自动化开发任务 | repo context、文件工具、shell、diff、测试闭环 | Agent 如何嵌入真实开发环境 |
| Codex | 本地 CLI / 云端任务 / GitHub | 代码任务委派、并行开发、PR 输出 | 云端环境、worktree、任务隔离、review | Agent 如何从助手变成后台执行者 |
```

这个表说明，不同 Agent 的核心学习点完全不同。

如果你想做个人 AI 员工，OpenClaw 和 Hermes 类助理的控制平面、渠道入口、长期记忆很值得看。

如果你想做代码 Agent，Claude Code 和 Codex 的 repo context、shell 工具、diff review、测试闭环非常重要。

如果你想做外贸 Agent，四者都能借鉴一部分：

```text
从 OpenClaw 学：多入口、控制平面、长期运行。
从 Hermes 类助理学：消息协作、角色身份、任务状态沟通。
从 Claude Code 学：工具权限、可审查输出、执行闭环。
从 Codex 学：任务委派、后台执行、并行任务隔离、review 流程。
```

这就是系统学习的价值。你不会被单个项目限制，而是能从多个项目中组合出适合自己场景的架构。

## 15.8 不要迷信闭源产品，也不要低估开源项目

研究 Claude Code、Codex 这类产品时，一个常见问题是：它们很多内部实现不可见。我们只能看到文档、外部行为和部分开源组件。那还有没有学习价值？

当然有，但要注意边界。

对于闭源产品，可以学习三类东西：

第一，产品形态。它在哪些界面出现？用户如何委派任务？如何 review？如何展示结果？这些都可以通过使用和官方文档观察。

第二，外部行为。它能读取代码库、编辑文件、运行命令、生成 diff、处理测试失败。这些行为本身能反推出它需要哪些机制，但不能等同于内部实现。

第三，交互设计。它何时请求确认？如何展示计划？如何处理失败？如何让用户接管？这些都是可学习的产品经验。

但不能把推断写成事实。例如不能说“Claude Code 内部一定使用某某状态机”或“Codex 一定使用某种 Planner”。除非官方文档或源码明确说明。

对于开源项目，问题相反。源码可见，但不要低估它的工程价值。有些开源项目虽然 UI 粗糙、文档不完美，但其中的插件系统、工具注册、任务日志、权限设计可能非常值得学习。

学习开源项目时也要注意：

```text
不要把历史包袱当最佳实践；
不要把项目特定结构当通用架构；
不要只看 star 数；
不要只看 README；
不要忽略 issue、PR、测试和文档；
不要因为能跑就认为设计成熟。
```

最好的方式仍然是第 14 章的方法：用机制框架读源码，用运行链路验证理解，用设计取舍判断是否复用。

## 15.9 从这些项目中提炼 12 个可迁移设计模式

综合 OpenClaw、Hermes 类助理、Claude Code、Codex，我们可以提炼出 12 个可迁移设计模式。

### 模式 1：自然语言入口 + 结构化工作台

聊天适合发起任务和快速纠偏，结构化界面适合查看复杂对象、审批和管理历史。不要把所有交互都塞进聊天框。

外贸 Agent 示例：用户用聊天说“找沙特客户”，但客户列表、评分、邮件草稿在表格工作台中审查。

### 模式 2：控制平面与执行平面分离

控制平面负责用户、权限、配置、审批、状态展示；执行平面负责模型、工具、任务运行。这样系统更容易扩展，也更安全。

### 模式 3：统一事件模型

无论入口来自 CLI、Web、Slack、邮件还是定时任务，内部都转换成统一事件。这样 runtime 不依赖具体渠道。

### 模式 4：任务委派而不是单轮问答

用户交给 Agent 的应该是任务，而不是每一步操作。系统要能创建 task、执行、暂停、恢复、完成和归档。

### 模式 5：可审查输出

Agent 的重要结果必须可 review。代码 Agent 展示 diff，外贸 Agent 展示客户来源和邮件草稿，教育 Agent 展示学习计划依据。

### 模式 6：工具风险分级

所有工具按风险分层。低风险自动，中风险记录，高风险审批，禁止类工具不开放。

### 模式 7：环境隔离

代码任务用 worktree 或 sandbox；业务任务用独立 task scope；多用户数据严格隔离。

### 模式 8：长期记忆可编辑

Agent 可以记住偏好和历史，但用户必须能查看、修改、删除。记忆不是黑箱。

### 模式 9：后台执行 + 前台通知

长任务不应该阻塞聊天。Agent 可以后台运行，在关键节点通知用户。

### 模式 10：失败可复盘

每次失败都应该留下 trace，说明看到了什么、做了什么、为什么失败。失败记录是系统迭代的重要资产。

### 模式 11：领域工具高于通用工具

`web_search` 很灵活，但 `search_potential_importers` 更适合外贸 Agent。领域工具能降低模型误用概率。

### 模式 12：Agent 是角色，不只是模型实例

一个 AI 员工应该有职责、权限、记忆、工具、评估标准和交付物格式。不要把多个角色混在一个万能 Agent 中。

## 15.10 如何把这些经验转化为自己的 Agent Runtime

学习这些项目后，最终要回到自己的系统。我们可以把可借鉴能力分成四层。

第一层是入口层。

```text
CLI
Web UI
聊天渠道
IDE 插件
定时任务
Webhook
```

不是所有入口都要一开始做，但内部要抽象成统一事件。

第二层是控制层。

```text
用户与权限
任务管理
审批队列
配置中心
通知系统
可观测面板
```

这层决定 Agent 是否可被真实用户控制。

第三层是运行时层。

```text
Agent Loop
Planner
Executor
Tool Registry
Context Manager
Memory Store
State Machine
Scheduler
```

这层是 Agent 的核心。

第四层是业务层。

```text
外贸获客
教育辅导
代码开发
运营分析
资料研究
企业流程
```

业务层定义角色、工具和交付物。

这四层组合起来，就是下一章要详细设计的 Agent Runtime。

## 15.11 资料依据与谨慎说明

本章涉及的公开资料应按“事实、推断、启发”三层使用。

可以作为公开事实引用的资料包括：

- OpenClaw GitHub README：说明 OpenClaw 是运行在自己设备上的个人 AI 助理，支持已有沟通渠道、语音能力和 live Canvas，并强调 Gateway 是控制平面，产品是 assistant。
- Claude Code 官方文档：说明 Claude Code 是 agentic coding tool，可以读取代码库、编辑文件、运行命令，并集成终端、IDE、桌面 app 和浏览器。
- OpenAI Codex web / cloud 文档：说明 Codex 可以读、改、运行代码，并可在云端后台、并行执行任务。
- OpenAI Codex CLI 文档：说明 Codex CLI 可以在本地终端运行，并在选定目录中读、改、运行代码。

这些资料可以支持“产品形态”和“公开能力”的描述，但不能支持对闭源内部实现的断言。比如，不能因为 Claude Code 能运行测试，就断言其内部一定有某个名为 `TestRunnerAgent` 的模块；不能因为 Codex cloud 能并行执行任务，就断言其内部一定采用某种具体队列系统。

本章中凡是涉及内部模块、状态机、权限系统、上下文组织方式的内容，除非有公开资料直接支持，都应理解为本书为了教学目的做出的工程推断或架构启发。读者在阅读成熟项目时，也应保持同样的习惯：事实要可查，推断要标注，启发要能迁移。

## 本章总结

OpenClaw、Hermes 类助理、Claude Code 和 Codex 分别代表了 Agent 系统的不同方向：个人助理、多渠道消息协作、代码环境嵌入、任务委派与云端执行。它们的价值不在于让我们复制表面功能，而在于帮助我们理解不同场景下 Agent 系统如何做取舍。

OpenClaw 类系统提醒我们，Agent 可以进入用户已有渠道，并通过控制平面管理长期助理。Hermes 类助理提醒我们，角色身份、消息沟通和任务状态对个人助理非常关键。Claude Code 提醒我们，代码 Agent 的核心是 repo context、工具权限、测试闭环和可审查输出。Codex 提醒我们，Agent 可以成为后台任务执行者，并通过环境隔离、并行任务和 review 流程完成复杂开发工作。

学习成熟项目的正确方式，是提炼模式，而不是搬运表面功能。自然语言入口 + 结构化工作台、控制平面与执行平面分离、统一事件模型、任务委派、可审查输出、工具风险分级、环境隔离、长期记忆可编辑，这些模式都可以迁移到自己的 Agent 系统中。

下一章，我们将把这些模式收敛成一个属于自己的 Agent Runtime 架构，明确每个模块的职责、接口和演进路线。
