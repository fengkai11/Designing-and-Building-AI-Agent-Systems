# 附录 G：6 个月项目路线

90 天路线解决“学会基本机制”的问题，6 个月路线解决“做出一个可持续迭代的 Agent 项目”的问题。这里以外贸客户开发 Agent 为主线，也可以替换成教育 Agent、代码 Agent 或 AI 员工 Athena。

## 第 1 个月：问题定义与最小闭环

目标：不要急着做平台，先证明一个闭环有价值。

任务：

1. 写产品合同；
2. 明确目标用户；
3. 明确任务输入和输出；
4. 建立最小 Agent Loop；
5. 手动准备 20 条样例数据；
6. 生成第一版结果。

交付物：

```text
product-contract.md
mvp-workflow.md
sample-data/
first-output.md
```

验收标准：

- 用户知道如何输入任务；
- 系统能产出有用结果；
- 人工能判断结果好坏；
- 至少发现 5 个真实问题。

## 第 2 个月：工具系统与数据结构

目标：把“能生成”变成“可管理”。

任务：

1. 定义核心数据模型；
2. 实现工具注册表；
3. 明确工具风险等级；
4. 实现客户去重、评分、草稿生成等核心工具；
5. 增加任务日志。

交付物：

```text
domain-model.md
tool-registry.md
lead-schema.py
trace-log.jsonl
```

验收标准：

- 每个结果都有来源；
- 每个动作都有日志；
- 高风险动作不会自动执行；
- 数据能复用。

## 第 3 个月：记忆、RAG 与业务知识库

目标：让系统开始积累业务资产。

任务：

1. 建立产品知识库；
2. 建立客户历史记录；
3. 建立用户偏好记忆；
4. 支持历史客户去重；
5. 支持基于产品资料生成邮件。

交付物：

```text
knowledge-base/
memory-store.db
customer-history.csv
rag-retriever.py
```

验收标准：

- 不重复开发已联系客户；
- 邮件内容能引用真实产品信息；
- 用户偏好能影响输出；
- 错误记忆可删除。

## 第 4 个月：审批、评估与可观测性

目标：让系统从 Demo 接近可用工具。

任务：

1. 实现审批队列；
2. 实现 trace 回放；
3. 建立评估集；
4. 统计成本、延迟和成功率；
5. 复盘失败案例。

交付物：

```text
approval-queue.md
eval-cases.yaml
eval-report.md
trace-viewer.md
```

验收标准：

- 邮件必须审批后才能发送；
- 失败任务可复盘；
- 每周能看到质量变化；
- 能识别主要失败类型。

## 第 5 个月：产品界面与人工协同

目标：从脚本变成可用产品。

任务：

1. 设计任务工作台；
2. 设计客户列表；
3. 设计审批页面；
4. 设计执行轨迹页面；
5. 设计设置页面。

交付物：

```text
frontend-spec.md
api-contract.md
review-dashboard.md
```

验收标准：

- 用户不看日志也能使用；
- 用户能修改任务方向；
- 用户能审批和拒绝动作；
- 用户能查看历史。

## 第 6 个月：小范围真实试用与迭代

目标：让真实用户使用，并根据反馈收敛产品。

任务：

1. 找 1–3 个真实使用者；
2. 每周运行固定任务；
3. 记录人工节省时间；
4. 记录有效客户数量；
5. 记录错误和风险；
6. 决定继续自用、外部销售还是开源。

交付物：

```text
pilot-report.md
user-feedback.md
pricing-hypothesis.md
next-version-plan.md
```

验收标准：

- 至少连续使用 4 周；
- 用户愿意继续试用；
- 找到明确高频痛点；
- 知道下一版该砍什么、补什么。

## 六个月后的三种方向

### 方向一：自用效率工具

适合已有业务场景，例如外贸工厂、拼多多店铺、内容运营。目标不是卖软件，而是提升自己业务效率。

### 方向二：垂直 SaaS

适合某个角色痛点明确、付费意愿强的场景。需要加强权限、多用户、数据隔离、稳定性和售后。

### 方向三：开源框架 + 闭源业务模块

适合构建 Athena 类系统。基础 Runtime 开源，行业数据、业务插件、部署服务和高级功能商业化。

## 长期原则

1. 不要追求“全自动”，追求“稳定提升人效”。
2. 不要先做大平台，先做高频闭环。
3. 不要让 Agent 直接接管高风险动作。
4. 不要只看生成质量，要看最终业务指标。
5. 不要只积累代码，要积累评估集和失败案例。
