从零构建 AI Agent 系统

第 3 章：什么时候该用 Agent，什么时候不该用 Agent

在前两章中，我们已经建立了两个基础认识。

第一，Agent 不是一个会聊天的模型，也不是一次工具调用，而是一个围绕目标持续执行任务的系统。第二，一个可用的 Agent 系统需要多层能力：任务理解、规划、工具、上下文、记忆、状态、评估、安全、人机协同和产品交互。

但是，理解 Agent 很重要，并不意味着所有任务都应该 Agent 化。

这是学习 Agent 的第三个关键问题：

什么时候该用 Agent，什么时候不该用 Agent？

这个问题比“如何写一个 Agent”更重要。因为很多失败的 Agent 项目，并不是技术实现写错了，而是一开始就把不该 Agent 化的任务强行 Agent 化了。结果系统变得更慢、更贵、更不稳定、更难排查，甚至比原来的普通自动化还差。

在真实工程中，一个好架构师不会看到大模型就想把所有流程都交给模型；一个好 Agent 开发者也不会看到工具调用就想让模型自由决策。真正成熟的判断是：先理解任务，再选择合适的自动化形态。

有些任务适合普通函数。有些任务适合确定性工作流。有些任务适合“工作流 + LLM 节点”。只有一部分任务真正适合 Agent。

本章的目标，就是建立这种判断力。

读完本章后，你应该能够面对一个真实需求，判断它应该用函数、脚本、工作流、RAG、LLM 节点，还是 Agent。你还应该能够说清楚：为什么这个任务需要或不需要 Agent；如果需要 Agent，应该让 Agent 负责哪些部分；如果不需要 Agent，使用 Agent 会带来哪些不必要的风险和成本。

这是一种非常基础但非常重要的能力。因为 Agent 系统的第一步，不是写代码，而是判断边界。

3.1 为什么“该不该用 Agent”是第一性问题

很多新技术刚出现时，开发者容易出现一种冲动：既然它很强，那就尽可能多地使用它。

大模型出现后，很多系统都被重新包装成“AI 驱动”。Agent 概念流行后，很多原本只是脚本、工作流或规则系统可以解决的问题，也被包装成 Agent。

比如：

用户输入订单号，系统查询物流状态。

这需要 Agent 吗？通常不需要。它只是一个明确的查询任务。后端根据订单号查数据库或调用物流 API，然后返回结果即可。让大模型规划、选择工具、反思结果，不仅没有必要，还可能增加延迟和成本。

再比如：

用户上传一张发票，系统识别金额、税号和日期，然后入库。

这需要 Agent 吗？也不一定。它可能需要 OCR、版面分析、字段校验和人工复核，但不一定需要 Agent。如果发票种类稳定、字段明确、流程固定，传统自动化就可以很好完成。只有当系统需要处理大量非标准文件、自动判断异常原因、主动请求补充材料、根据业务规则决定下一步时，才可能引入 Agent 或 LLM 节点。

再看一个更复杂的任务：

用户说：

帮我分析最近 30 个潜在客户的回复，判断哪些值得继续跟进，哪些应该放弃，并为每个客户生成下一步动作。

这个任务就更接近 Agent。因为它不是简单查询，也不是固定表单处理。它需要理解客户回复内容，结合历史沟通记录，判断客户意向，制定跟进策略，并可能生成不同类型的回复草稿。这里存在开放判断、多步骤处理、上下文依赖和业务目标优化。

这三个例子的区别说明：是否使用 Agent，不取决于任务看起来是否“智能”，而取决于任务的结构。

一个任务是否适合 Agent，至少要看六个方面：

第一，它是否是目标驱动，而不是单步指令。

第二，它是否需要多步骤执行，而不是一次函数调用。

第三，它是否存在不确定性，需要模型在执行过程中判断下一步。

第四，它是否需要使用外部工具，例如搜索、文件、数据库、浏览器、代码执行、消息系统等。

第五，它是否需要状态、记忆或长期跟踪。

第六，它的错误后果是否可以通过边界设计控制。

如果一个任务只有第一点而没有后面几点，它可能只是一个普通问答。如果一个任务有工具调用但没有多步骤和不确定性，它可能只是 Tool Calling。如果一个任务有多步骤但步骤完全固定，它可能更适合工作流。如果一个任务同时具备目标驱动、多步骤、不确定性、工具使用和状态跟踪，它才更适合 Agent。

这就是本章的核心判断逻辑。

3.2 函数、脚本、工作流、LLM 节点和 Agent

为了判断什么时候该用 Agent，我们必须先区分几种不同的自动化形态。

很多时候，开发者不是在“用不用 AI”之间做选择，而是在不同层级的自动化方案之间做选择。它们分别是函数、脚本、工作流、LLM 节点和 Agent。

3.2.1 函数：确定输入到确定输出

函数适合解决边界明确、输入输出清楚、逻辑稳定的问题。

例如：

price = unit_price * quantity

这是最简单的函数。输入单价和数量，输出总价。这个过程不需要大模型，也不需要 Agent。

再比如：

def calculate_profit(revenue, cost, ad_spend):
    return revenue - cost - ad_spend

这类计算任务应该用函数完成。即使你让大模型来算，它也可能因为格式、单位或四舍五入出错。确定性计算应该交给确定性程序。

在外贸场景中，计算 FOB 报价、毛利率、汇率换算、包装体积、集装箱装载数量等，如果公式清楚，都应该优先用函数，而不是 Agent。

例如：

计算 5000 个钢卷尺的总重量和装箱体积。

只要产品规格和包装参数明确，这就是函数或普通业务逻辑，不需要 Agent。Agent 可以帮助解释结果，但不应该代替确定性计算。

3.2.2 脚本：一组固定步骤

脚本适合执行一组固定步骤，通常用于批处理、数据转换、文件处理、自动导出等任务。

例如：

每天凌晨从服务器下载日志，解压，过滤错误行，生成统计表。

这是典型脚本任务。它有步骤，但步骤固定；它可能有异常处理，但不需要模型自己决定下一步。

在电商运营中，脚本可以完成：

从平台导出订单 CSV；

清洗字段；

统计每个 SKU 的销量；

生成日报表。

如果规则明确，用 Python、SQL 或定时任务即可。没有必要让 Agent 每次重新“思考”如何处理 CSV。

3.2.3 工作流：稳定流程中的状态控制

工作流适合多步骤、跨系统、需要状态管理但流程相对稳定的任务。

例如，一个报销流程：

员工提交报销单；

系统校验发票；

部门经理审批；

财务复核；

打款；

归档。

这个流程不应该让 Agent 自由决定顺序。因为审批顺序、权限和状态都必须严格控制。工作流引擎可以保证每一步按规则发生。

在外贸业务中，一个报价流程可能是：

客户询盘；

业务员确认需求；

系统匹配产品规格；

计算报价；

经理审核价格；

发送报价单；

记录客户反馈。

这里的主流程也更适合工作流。Agent 可以嵌入其中，例如帮业务员总结客户需求、生成报价说明、识别异常条款，但不应该完全替代流程控制。

3.2.4 LLM 节点：工作流中的智能处理环节

很多任务不需要完整 Agent，但适合在工作流中加入 LLM 节点。

所谓 LLM 节点，就是在一个确定流程的某个步骤中，调用大模型完成理解、分类、生成、摘要或判断。

例如，客服系统中有一个流程：

收到客户消息；

识别客户意图；

分配到对应部门；

生成回复草稿；

人工确认后发送。

这里，“识别客户意图”和“生成回复草稿”可以用 LLM，但整个流程仍然是工作流，不一定需要 Agent 自主规划。

再比如，教育系统中，学生提交一道错题。系统可以调用 LLM 节点生成错因分析：

错误类型：公式使用错误
涉及知识点：一元二次方程求根公式
建议练习：基础公式代入题 5 道

这个 LLM 节点很有用，但它只是一个节点。它不需要自己决定今天给学生安排什么长期计划，也不需要自动通知老师，除非系统进一步设计成 Agent 或工作流。

3.2.5 Agent：围绕目标进行开放执行

Agent 适合目标驱动、多步骤、不确定、需要工具和状态的任务。

例如：

每天帮我寻找 20 个适合钢卷尺出口的潜在客户，要求优先找五金批发商和工程用品分销商，排除零售小店，生成评分和开发信草稿，并避免重复开发。

这个任务不只是一个工作流。因为“寻找客户”本身是不确定的，搜索关键词需要调整，网站质量需要判断，客户类型需要识别，重复客户需要排除，开发信需要根据客户画像定制。

当然，它也不能完全自由运行。更好的架构是“工作流控制主流程，Agent 处理开放判断”。比如：

搜索候选客户这个环节由 Agent 执行；

客户去重由规则和数据库执行；

客户评分由规则 + LLM 判断执行；

邮件草稿由 LLM 生成；

发送前必须人工审批。

这说明，Agent 不是要取代函数、脚本和工作流，而是和它们协作。

一个成熟系统里，确定性部分应该尽量确定，开放性部分才交给 Agent。

3.3 判断任务是否适合 Agent 的六个维度

接下来，我们把判断标准系统化。

面对一个任务，可以从六个维度分析。

3.3.1 目标开放性

第一个维度是目标是否开放。

如果用户目标非常明确，比如：

查询订单 12345 的物流状态。

这不是开放目标。系统只需要查询并返回。

如果用户目标是：

帮我判断这批客户里哪些值得重点跟进。

这就是开放目标。因为“值得重点跟进”需要结合客户规模、回复意向、产品匹配度、地区、历史沟通、报价阶段等因素判断。

开放目标通常有几个特点：

用户只描述结果，不描述路径；

达成目标的方法不止一种；

中间需要做判断；

结果质量存在主观评价；

执行过程中可能需要调整策略。

例如“帮我写一篇文章”本身是开放目标，但未必需要 Agent。因为它虽然开放，但不一定需要多步骤工具执行。如果只是根据已有信息写作，Chatbot 就够了。如果需要调研、检索、引用、对比、更新资料、保存版本，那才更接近 Agent。

所以目标开放性是必要条件之一，但不是充分条件。

3.3.2 步骤不确定性

第二个维度是步骤是否不确定。

固定步骤适合工作流，不确定步骤适合 Agent。

例如：

用户上传合同，系统提取甲方、乙方、金额、日期，然后生成摘要。

如果合同格式相对稳定，这可以是工作流加 LLM 节点。步骤很固定：上传、解析、提取、摘要、审核。

但如果任务是：

帮我审查这份合同中有哪些潜在风险，并判断哪些条款需要和对方重新谈。

这就不只是固定字段提取。Agent 或 LLM 节点需要根据合同内容识别风险点，可能需要查公司内部政策、历史合同、法律模板，还可能需要提出修改建议。如果再加入多轮审查、资料检索和协同修改，就更接近 Agent。

步骤不确定性越高，Agent 的价值越大。但不确定性也意味着风险和不可控性更高，因此需要更强的边界设计。

3.3.3 外部工具依赖

第三个维度是是否需要外部工具。

如果任务只需要模型根据当前输入回答，通常不需要 Agent。例如：

请解释一下什么是向量数据库。

这是问答。

如果任务需要实时信息、文件、数据库、浏览器、代码执行或业务系统，就需要工具。

但有工具并不等于需要 Agent。

例如：

查询今天美元兑人民币汇率。

这是工具调用，但不需要 Agent。

而下面这个任务更接近 Agent：

根据最近三个月汇率波动、目标客户所在国家、我们的报价周期，帮我制定一个更稳妥的外贸报价策略。

这里可能需要查询汇率、读取历史报价、分析客户国家、总结风险，再生成策略。工具调用只是其中一部分，关键在于多步骤判断。

工具依赖越多，Agent 越可能有用，但工具越多，也越需要权限控制。

3.3.4 状态持续性

第四个维度是任务是否需要持续状态。

一次性任务可以用 Chatbot、Tool Calling 或工作流完成。长期任务更适合 Agent 或工作流 + Agent。

例如：

总结这篇文章。

这是一次性任务。

每周追踪这个行业的新论文，筛选与机器人操作相关的内容，更新我的研究列表。

这是持续任务。它需要知道上周已经看过哪些论文，哪些已加入列表，哪些被排除，用户偏好哪些方向，是否有重要更新。

外贸客户开发也是典型持续任务。系统必须知道：

哪些客户已经发现；

哪些客户已经联系；

哪些客户回复了；

哪些客户不再跟进；

哪些客户需要下周提醒；

哪些开发信模板效果更好。

没有状态持续性，Agent 很容易重复工作，甚至造成业务混乱。

3.3.5 错误后果

第五个维度是错误后果。

如果一个任务失败只是生成结果质量不高，风险较低。例如：

帮我生成 10 个标题。

模型生成得不好，用户可以重试。

但如果任务会对真实世界产生影响，风险就高。例如：

发送邮件；

提交代码；

删除文件；

修改数据库；

下单付款；

向客户报价；

修改生产计划。

高风险任务不是不能用 Agent，而是不能让 Agent 完全自主。必须加入审批、权限、回滚、日志和人工确认。

例如，外贸 Agent 可以自动生成开发信，但不应该默认自动发送。代码 Agent 可以生成修改计划和 diff，但不应该默认直接 push 到主分支。

判断是否使用 Agent 时，不仅要看能力需求，还要看错误后果是否可控。

3.3.6 结果可验证性

第六个维度是结果是否可验证。

Agent 输出结果后，系统或人类能否判断它是否正确？

例如，计算利润率很容易验证。代码测试结果也相对容易验证。客户名单质量就较难验证，因为客户是否真实有价值，可能要等触达之后才知道。

结果越难验证，越需要保守设计。

比如让 Agent 自动总结文章，用户可以快速检查。让 Agent 自动判断投资决策，就高风险且难验证。让 Agent 自动给学生打标签，也需要谨慎，因为错误画像可能影响后续学习安排。

一个任务如果既开放、又高风险、又难验证，就不能简单交给 Agent。即使引入 Agent，也应该把它定位为辅助分析，而不是最终决策者。

3.4 一个实用判断表：函数、工作流还是 Agent

为了让判断更直观，我们可以建立一个简单表格。

任务特征更适合的方案
输入输出明确，逻辑固定函数
多个固定步骤，顺序稳定脚本或工作流
流程稳定，但某一步需要理解或生成工作流 + LLM 节点
需要实时查询，但步骤简单Tool Calling
目标开放，多步骤，不确定，需要工具Agent
长期运行，需要状态、记忆和调度Long-running Agent 或工作流 + Agent
高风险动作较多Agent + 审批 + 权限控制
结果难验证且后果严重谨慎使用，最好只做辅助建议

这个表格不是绝对规则，但可以作为工程判断起点。

我们可以用几个例子来测试。

例子一：计算拼多多产品毛利

任务：

输入产品售价、进货价、平台扣点、推广费、快递费，计算毛利。

适合方案：函数或表格公式。

原因：输入输出明确，公式确定。即使后面做成经营分析系统，毛利计算本身也不该由 Agent 自由生成。

Agent 可以做什么？

Agent 可以解释毛利结果，例如提醒“当前推广费占售价比例过高，停推广后可能无单”。但计算本身应由确定性逻辑完成。

例子二：生成开发信

任务：

根据客户公司信息和产品卖点，生成一封英文开发信。

适合方案：LLM 节点。

原因：这是生成任务，但不一定需要 Agent。输入清楚，输出是一封邮件草稿。

什么时候需要 Agent？

如果任务变成：

找客户、判断客户类型、生成画像、选择卖点、写开发信、安排跟进。

这才更接近 Agent。

例子三：自动整理客户回复

任务：

每天读取客户邮件回复，判断意向，生成跟进建议。

适合方案：工作流 + LLM 节点，进一步可发展为 Agent。

原因：每天处理邮件是固定流程，但理解回复、判断意向和生成建议需要 LLM。如果还要根据不同回复自动安排任务、更新 CRM、触发提醒，则更接近 Agent。

例子四：让 AI 自主经营一个店铺

任务：

让 Agent 自己选品、上架、定价、推广、处理客服、优化投放。

适合方案：不能直接全 Agent 化。

原因：这个任务过大、风险高、涉及资金、平台规则、供应链和客户体验。更合理的做法是拆成多个子系统：

选品分析：Agent 辅助研究；

毛利计算：函数；

上架流程：工作流；

标题生成：LLM 节点；

推广预算建议：Agent 辅助分析；

实际投放调整：人工确认。

这说明，宏大目标通常不能直接交给单个 Agent，而要拆成可控模块。

例子五：代码仓库 bug 修复

任务：

阅读项目，定位 bug，修改代码，运行测试，生成 diff。

适合方案：Agent，但需要权限控制。

原因：它需要理解 repo、搜索文件、制定计划、修改代码、运行测试、根据失败结果调整。步骤不完全固定，工具依赖强，状态持续性明显。

风险在哪里？

写文件、执行 shell、删除文件、提交代码都可能有风险。因此要设置只读分析、修改计划确认、测试命令白名单、diff 审核和回滚机制。

3.5 不该用 Agent 的典型场景

理解什么时候不该用 Agent，往往比知道什么时候该用更重要。

3.5.1 确定性计算任务

所有确定性计算，应该优先用代码、公式或数据库查询完成。

例如：

汇率换算；

毛利计算；

库存扣减；

订单金额汇总；

ROI 计算；

排班工时统计；

几何坐标变换；

相机投影公式计算。

这些任务的核心要求是准确和可重复。大模型擅长解释和推理，但不应该替代确定性程序。

比如在自动驾驶感知中，点云投影到图像平面需要严格的外参、内参和畸变模型。这个过程不能交给 Agent “感觉一下方向对不对”。Agent 可以帮助检查代码、解释公式、生成调试流程，但最终投影计算必须由确定性数学实现。

3.5.2 规则明确的审批流程

如果流程规则明确，应该用工作流。

例如：

员工请假审批；

报销流程；

合同盖章流程；

订单发货流程；

生产排期流程。

Agent 可以辅助其中某些环节，例如总结合同风险、识别报销异常、生成审批意见，但不能让 Agent 自由决定流程顺序。

3.5.3 高频低价值任务

Agent 调用模型和工具有成本。如果一个任务高频、简单、低价值，使用 Agent 可能不划算。

例如：

每次用户打开页面都让 Agent 分析一遍展示顺序；

每条普通日志都让 Agent 判断是否异常；

每个简单客服问题都启动多步 Agent。

这类任务更适合规则、缓存、分类模型或轻量 LLM 节点。Agent 应该用于价值足够高、复杂度足够高的任务。

3.5.4 需要强一致性的核心交易

涉及资金、库存、权限和核心数据一致性的任务，不应由 Agent 直接决策执行。

例如：

自动扣款；

自动退款；

自动修改账户权限；

自动删除生产数据；

自动提交财务报表。

这些操作需要严格规则、权限、审计和人工确认。Agent 可以提供建议，但不能作为最终执行者。

3.5.5 法律、医疗、金融等高风险最终决策

在高风险领域，Agent 更适合作为辅助工具，而不是最终决策者。

例如：

医疗诊断；

法律意见；

投资决策；

贷款审批；

保险拒赔；

劳动纠纷处理。

这些场景可以使用 Agent 做资料整理、初步分析、案例检索、风险提示，但最终判断必须由专业人员负责，并且系统要明确限制责任边界。

3.5.6 用户意图极其模糊且缺少反馈渠道

如果用户目标模糊，但系统没有澄清机制，也不允许中途反馈，Agent 容易跑偏。

例如用户只说：

帮我赚钱。

这不是一个可执行任务。Agent 不能直接开始乱搜索、乱决策。它应该先澄清目标：预算、周期、风险承受能力、资源、行业、能力、限制条件等。

如果系统没有澄清机制，这类任务不适合直接执行。

3.6 适合 Agent 的典型场景

接下来，我们看哪些任务真正适合 Agent。

3.6.1 开放式研究任务

研究任务通常适合 Agent，尤其当它需要多轮检索、资料判断和报告生成时。

例如：

调研当前工业界对机器人 VLA 的落地情况，区分学术研究、开源项目、商业产品和仍未解决的问题。

这个任务需要搜索资料、比较来源、组织观点、判断不确定性。它不是简单问答，也不是一次工具调用。Agent 可以先拆解问题，再检索多个来源，然后生成结构化报告。

但研究 Agent 也要有边界。它应该标注来源，区分事实和推断，说明不确定性，避免编造引用。

3.6.2 信息发现和筛选任务

Agent 很适合在开放网络或复杂数据中发现候选对象，并进行初步筛选。

例如：

寻找潜在客户；

筛选供应商；

寻找竞品；

筛选招聘候选人；

寻找论文；

寻找开源项目。

这些任务的共同特点是：目标对象不在单一数据库里，搜索路径不固定，结果质量需要判断。

外贸获客就是典型例子。Agent 可以调整关键词、访问多个来源、提取信息、判断公司类型、生成评分。

3.6.3 多步骤内容生产

普通内容生成不一定需要 Agent，但多步骤内容生产适合 Agent。

例如：

帮我写一本关于 Agent 系统构建的技术书。先生成目录，再逐章写作，每章保持风格一致，最后整理成 Markdown 文档包。

这个任务就不只是一次写作。它需要全书规划、章节生成、风格一致性、文件管理、版本打包和持续推进。

如果只是写一篇短文，Chatbot 就够了；如果是长期书稿项目，Agent 式的任务管理和状态保持就有价值。

3.6.4 代码理解和修改任务

代码任务通常非常适合 Agent，因为它天然需要工具和多步骤执行。

例如：

阅读 repo；

定位入口；

搜索相关文件；

理解调用链；

生成修改计划；

编辑代码；

运行测试；

根据错误修复；

生成 diff。

但代码 Agent 必须受到限制。只读分析可以自动执行，写入文件需要计划，危险 shell 命令要禁止，提交代码要人工确认。

3.6.5 长期监督和提醒任务

如果任务需要长期运行、定期检查、状态更新，Agent 会很有价值。

例如：

每天检查目标客户是否有新动态；

每周总结学生错题变化；

每月分析店铺经营数据；

持续跟踪某个技术方向的新论文；

监控竞争对手网站变化。

这些任务不仅需要定时触发，还需要理解变化、判断重要性、生成建议。普通定时脚本可以抓数据，但 Agent 可以解释变化和提出下一步动作。

3.6.6 人类助理型任务

很多助理型任务适合 Agent，因为它们本来就不是单步操作，而是围绕目标持续推进。

例如：

安排会议；

整理资料；

跟进邮件；

准备报告；

管理待办；

协调多个系统。

但助理型 Agent 的关键是权限和确认。它可以建议会议时间，但不应随意替用户发送敏感邮件；它可以整理待办，但不应删除用户任务；它可以生成回复，但发送前应确认。

3.7 Agent 适用性评分模型

为了更系统地判断，我们可以给任务做一个简单评分。

每个维度 0 到 2 分：

维度0 分1 分2 分
目标开放性明确单步部分开放高度开放
步骤不确定性固定步骤少量分支需要探索
工具依赖不需要工具需要少量工具需要多工具协同
状态持续性一次性任务内状态跨任务长期状态
判断复杂度规则判断简单语义判断复杂业务判断
结果可验证性容易验证部分可验证难验证

注意，结果可验证性这一项不是越高越适合 Agent。这里的 2 分表示结果难验证，意味着需要更谨慎设计。如果前五项高分，而结果可验证性也高分，说明这个任务可能需要 Agent，但必须加强评估和人工审查。

我们可以用这个模型分析几个任务。

任务一：订单物流查询

目标开放性：0

步骤不确定性：0

工具依赖：1

状态持续性：0

判断复杂度：0

结果可验证性：0

结论：不需要 Agent。Tool Calling 足够。

任务二：外贸客户开发

目标开放性：2

步骤不确定性：2

工具依赖：2

状态持续性：2

判断复杂度：2

结果可验证性：1 或 2

结论：适合 Agent，但必须有审批、去重、评分解释、触达记录和人工确认。

任务三：商品标题生成

目标开放性：1

步骤不确定性：0

工具依赖：0 或 1

状态持续性：0

判断复杂度：1

结果可验证性：1

结论：通常用 LLM 节点即可，不需要完整 Agent。

任务四：代码仓库 bug 修复

目标开放性：2

步骤不确定性：2

工具依赖：2

状态持续性：1

判断复杂度：2

结果可验证性：1

结论：适合 Agent，但要有测试、diff、权限和回滚。

任务五：学生每日错题推荐

目标开放性：1

步骤不确定性：1

工具依赖：1

状态持续性：2

判断复杂度：2

结果可验证性：1

结论：适合工作流 + Agent。每日任务可以由工作流触发，错因分析和推荐策略可由 Agent 或 LLM 节点完成。

这个评分模型不是为了给出机械答案，而是帮助你把任务拆开看。很多时候，系统不是“整体用 Agent”或“整体不用 Agent”，而是某些子环节适合 Agent。

3.8 把大任务拆成可控子任务

很多需求一开始都太大，不能直接判断是否适合 Agent。正确做法是拆分。

例如：

帮我做一个能自动经营拼多多店铺的 Agent。

这是一个巨大目标。如果直接做成 Agent，非常危险。因为它涉及选品、定价、上架、客服、推广、库存、供应链、售后、资金和平台规则。

我们应该把它拆成子任务：

  1. 竞品数据收集；
  2. 商品毛利计算；
  3. 标题和主图文案生成；
  4. 上架信息填写；
  5. 推广数据分析；
  6. 客服回复草稿；
  7. 差评原因总结；
  8. 库存预警；
  9. 经营日报生成；
  10. 调价建议。

然后分别判断。

毛利计算适合函数。

库存预警适合规则。

经营日报适合脚本 + LLM 总结。

客服回复草稿适合 LLM 节点 + 人工确认。

竞品分析可能适合 Agent，因为它需要收集、对比、判断和总结。

推广策略建议可能适合 Agent 辅助，但实际投放调整需要人工确认。

这样拆开后，系统会更稳。

同样，外贸 Agent 也应该拆开：

客户搜索；

客户识别；

客户去重；

客户评分；

开发信生成；

邮件审批；

发送记录；

回复理解；

跟进计划。

其中，去重可以用规则和数据库；评分可以用规则 + LLM；开发信生成可以用 LLM；客户搜索和识别更适合 Agent；邮件发送必须审批。

代码 Agent 也可以拆开：

repo 结构扫描；

需求理解；

相关文件搜索；

修改计划生成；

代码编辑；

测试执行；

错误修复；

diff 总结。

其中，测试执行是工具，diff 是确定性输出，错误日志理解适合 LLM，整个循环适合 Agent。

这个拆分过程非常重要，因为它能避免“一个 Agent 管所有事”的冲动。

一个可控系统通常不是一个超级 Agent，而是一组确定性模块、LLM 节点和 Agent 节点的组合。

3.9 Agent 边界设计：让 Agent 做它擅长的部分

判断是否使用 Agent 后，还要进一步判断：Agent 负责哪些部分？

一个成熟设计不会简单写：

让 Agent 完成整个任务。

而会明确：

Agent 输入是什么；

Agent 可以使用哪些工具；

Agent 可以自动执行哪些动作；

Agent 哪些动作需要审批；

Agent 输出什么格式；

Agent 什么时候停止；

Agent 失败后如何处理；

人类在哪里介入。

例如，外贸客户开发 Agent 可以这样划边界：

Agent 可以自动执行：

搜索网页；

提取公开公司信息；

生成客户画像；

计算初步评分；

生成邮件草稿。

Agent 需要人工确认：

是否把客户加入正式 CRM；

是否发送开发信；

是否对某个客户继续跟进；

是否修改报价策略。

Agent 禁止执行：

自动群发邮件；

伪造联系人；

编造客户采购意向；

绕过平台规则爬取数据；

向客户承诺未经确认的价格和交期。

这样设计后，Agent 的价值仍然很大，但风险被控制住了。

代码 Agent 也可以这样划边界：

Agent 可以自动执行：

读取文件；

搜索代码；

生成修改计划；

运行安全白名单中的测试命令；

生成 diff 摘要。

Agent 需要人工确认：

批量修改文件；

安装新依赖；

执行数据库迁移；

提交代码；

删除文件。

Agent 禁止执行：

访问敏感凭证；

执行未知远程脚本；

删除项目目录；

修改生产配置；

绕过测试直接提交。

边界设计的本质，是把 Agent 的自主性限制在可接受范围内。

这也是 Agent 系统和普通聊天机器人的一个重大区别。聊天机器人错了，最多是回答错。Agent 错了，可能会行动错。因此，边界必须成为设计的一部分。

3.10 工作流优先原则

在工程实践中，可以遵循一个非常实用的原则：

能用函数解决的，不要用 LLM；能用工作流解决的，不要用 Agent；只有开放、不确定、需要判断的部分，才交给 Agent。

这个原则不是保守，而是工程理性。

函数更快、更便宜、更稳定。

工作流更可控、更可审计、更容易测试。

Agent 更灵活，但也更慢、更贵、更难预测。

因此，一个合理系统应该尽量把确定性部分收敛到函数和工作流，把不确定部分隔离成 Agent 节点。

以“生成外贸报价单”为例。

用户说：

客户要 5000 个 5 米钢卷尺，目标港口迪拜，帮我生成报价。

系统可以这样分工：

产品规格匹配：数据库查询；

基础价格计算：函数；

运费估算：调用物流 API；

毛利校验：函数；

报价单格式生成：模板；

英文说明文本：LLM 节点；

异常条款识别：Agent 或 LLM；

最终发送：人工确认。

如果让 Agent 从头到尾自由报价，风险很大。它可能算错价格，忘记 MOQ，忽略汇率，承诺错误交期，甚至生成不符合公司政策的付款条款。

但是，如果完全不用 LLM，系统又可能很死板，无法处理客户的自然语言需求。

所以最优方案通常是混合架构。

再看教育场景。

系统给学生安排复习任务时：

知识点映射可以用规则；

错题统计可以用数据库；

间隔复习可以用算法；

错因解释可以用 LLM；

综合学习计划可以用 Agent 辅助；

最终重要干预由老师确认。

这就是工作流优先、Agent 补充开放判断的设计思想。

3.11 什么时候要从 LLM 节点升级为 Agent

很多系统一开始不需要 Agent，只需要 LLM 节点。但随着需求发展，可能逐步升级。

判断是否需要升级，可以看几个信号。

信号一：一次模型调用不够了

如果一个任务用一次 LLM 调用就能完成，例如摘要、改写、分类，就不需要 Agent。

但如果它需要多轮模型调用，每一轮依赖上一轮结果，并且中间要调用不同工具，就可能需要 Agent。

例如，客户回复分类最初可以是一次调用：

输入邮件，输出意向等级。

后来你发现还需要读取历史沟通、查看报价、判断客户国家、生成跟进动作、创建提醒任务。此时单个 LLM 节点已经不够，可能需要 Agent 或工作流 + Agent。

信号二：步骤顺序需要动态调整

如果步骤固定，工作流即可。

但如果系统需要根据中间结果决定下一步，例如搜索不到客户就换关键词，找到零售店就换渠道，发现联系人缺失就去其他来源补充，那么 Agent 更合适。

信号三：需要长期记忆

如果任务只处理当前输入，不需要记忆，LLM 节点足够。

但如果系统需要记住用户偏好、历史结果、失败原因和长期状态，就需要更接近 Agent 的架构。

例如，开发信生成一开始只是 LLM 节点。后来系统要记住哪些客户已联系、哪种邮件模板回复率高、哪些国家适合哪种卖点，就需要记忆系统和长期任务管理。

信号四：需要主动推进任务

LLM 节点通常是被动调用。Agent 可以主动推进。

例如，用户每次手动输入“帮我总结这封邮件”，这是 LLM 节点。

如果系统每天自动读取新邮件，判断是否需要跟进，生成待办，并提醒用户审批，这就是 Agent 化方向。

信号五：需要处理失败和恢复

如果任务失败后只需要返回错误，普通工具调用即可。

但如果失败后需要分析原因、尝试替代路径、恢复状态、继续执行，就适合 Agent。

例如，搜索客户时某个网站打不开，Agent 可以换搜索源；代码测试失败时，Agent 可以读取错误日志并修复；RAG 检索不到答案时，Agent 可以改写查询。

3.12 什么时候要从 Agent 降级为工作流

有些系统一开始设计成 Agent，后来应该降级。

这听起来反直觉，但很常见。

当一个 Agent 反复执行同类任务后，你会发现其中很多步骤其实已经稳定。这时应该把稳定步骤固化成规则或工作流，而不是每次都让模型重新思考。

例如，外贸 Agent 一开始需要探索客户搜索策略。经过一段时间后，你发现：

对于阿联酋市场，某些关键词效果稳定；

某些目录站质量很差，应直接排除；

客户评分规则已经比较固定；

开发信结构已经稳定。

这时，可以把这些经验固化：

固定排除规则；

固定评分字段；

固定邮件模板；

固定审批流程。

Agent 只负责新情况、异常情况和复杂判断。

这就是从 Agent 到工作流的沉淀过程。

一个成熟 Agent 系统应该不断把重复成功的行为沉淀为规则、模板、工具和流程。否则，系统会一直昂贵且不稳定。

代码 Agent 也是如此。刚开始，Agent 需要探索如何运行测试。后来发现项目固定使用 pytest，就可以把测试命令固化。Agent 不需要每次猜。

教育 Agent 也是如此。某些错题推荐策略成熟后，可以变成规则；只有复杂错因分析和个性化解释继续由 LLM 处理。

所以，Agent 不是最终形态。Agent 可以作为探索层，而工作流和规则是沉淀层。

这也是优秀系统持续进化的方式：

让 Agent 探索不确定性；

把稳定经验沉淀为确定性；

让系统越来越便宜、稳定、可控。

3.13 一个完整案例：外贸客户开发任务拆解

现在我们用一个完整案例，把本章方法串起来。

用户需求：

帮我开发沙特和阿联酋的钢卷尺客户，优先找五金批发商、工具分销商和工程用品采购商，排除零售小店，生成客户名单、评分和开发信。

第一步，判断任务属性。

目标开放性高。用户只给了目标，没有给路径。

步骤不确定性高。搜索渠道、关键词、客户判断都需要动态调整。

工具依赖高。需要搜索、网页读取、表格存储、邮件草稿生成。

状态持续性中到高。客户不能重复开发，后续还要跟进。

判断复杂度高。客户类型和价值需要综合判断。

错误后果中到高。如果自动发邮件或编造信息，会影响业务。

结论：适合 Agent，但必须有边界。

第二步，拆分子任务。

子任务推荐方案
生成搜索关键词Agent 或 LLM 节点
搜索候选公司Agent + Search Tool
提取公司信息Tool + LLM 节点
判断客户类型LLM 节点 + 规则
去重函数 + 数据库
商机评分规则 + LLM
生成开发信LLM 节点
邮件发送人工审批后执行
触达记录工作流 + 数据库
后续跟进Scheduler + Agent

第三步，划定边界。

Agent 可以自动搜索和整理公开信息。

Agent 可以提出评分，但必须给出依据。

Agent 可以生成邮件草稿，但不能自动发送。

Agent 不能编造联系人、采购需求或客户意向。

Agent 不能绕过网站规则或高频爬取。

Agent 不能承诺价格、交期和付款条款，除非来自已确认产品知识库。

第四步，设计输出。

客户名单不能只是公司名，而应包括：

公司名称；

官网；

国家和城市；

客户类型；

主营业务；

联系方式；

信息来源；

匹配理由；

风险提示；

商机评分；

推荐下一步。

第五步，设计停止条件。

比如：

找到 30 个候选客户后停止；

连续 5 次搜索无有效结果后停止；

达到成本上限后停止；

遇到需要用户选择方向时暂停；

生成邮件草稿后进入审批队列。

第六步，设计长期记忆。

系统应记录：

已经发现的客户；

已经排除的客户；

已联系客户；

客户回复；

用户偏好的客户类型；

高质量搜索关键词；

低质量渠道黑名单。

这样，这个任务才从“让模型帮我找客户”变成一个可控的 Agent 系统。

3.14 一个完整案例：代码开发任务拆解

再看代码 Agent。

用户需求：

帮我给现有项目增加登录功能，支持邮箱验证码登录，并更新 README。

第一步，判断任务属性。

目标开放性高。用户没有说明项目结构和实现方式。

步骤不确定性高。需要读 repo 后决定修改哪些文件。

工具依赖高。需要读取文件、搜索、写代码、运行测试。

状态持续性中等。任务内需要记住已读文件和修改计划。

判断复杂度高。需要理解架构和避免破坏现有功能。

错误后果高。错误代码可能破坏项目。

结论：适合 Agent，但必须采用分阶段审批。

第二步，拆分：

子任务推荐方案
扫描目录结构工具
识别技术栈LLM 节点
搜索相关文件Agent
生成修改计划Agent
用户确认计划人工审批
修改代码Agent + 文件工具
运行测试工具，命令白名单
修复测试失败Agent
生成 diff工具
最终提交人工确认

第三步，边界：

可以自动读取文件；

可以自动生成计划；

写文件前最好展示计划；

不能执行危险 shell；

不能访问密钥；

不能直接 push；

删除文件必须确认；

数据库迁移必须确认。

第四步，停止条件：

计划生成后暂停；

修改完成并测试通过后停止；

连续 3 次测试失败后暂停请求用户帮助；

发现需求冲突时暂停澄清。

第五步，评估：

测试是否通过；

diff 是否符合计划；

是否修改了无关文件；

README 是否更新；

是否引入危险依赖；

是否保留可回滚点。

通过这个拆解可以看出，代码 Agent 的核心不是“模型会写代码”，而是“系统能控制代码修改过程”。

3.15 一个完整案例：教育 Agent 任务拆解

教育 Agent 适合说明长期状态和人机协同。

用户需求：

帮老师管理学生数学学习，记录错题，分析薄弱点，安排复习，并提醒老师干预。

第一步，判断任务属性。

目标开放性中到高。提升学习效果是长期目标。

步骤不确定性中等。错题分析和复习安排需要判断。

工具依赖中等。需要题库、学生记录、消息系统。

状态持续性高。学生画像要长期维护。

判断复杂度高。错因和学习状态需要解释。

错误后果中等到高。错误画像可能影响学习安排。

结论：适合工作流 + Agent，而不是完全自主 Agent。

第二步，拆分：

子任务推荐方案
记录做题结果数据库
统计错题知识点函数 / SQL
错因分析LLM 节点
学生画像更新规则 + LLM + 人工可修正
每日练习推荐工作流 + 推荐规则
复杂学习计划Agent 辅助
老师干预提醒工作流
家长报告生成LLM 节点 + 审核

第三步，边界：

AI 可以分析错因，但应允许老师修改；

AI 可以推荐练习，但不能过度增加负担；

AI 可以提醒老师，但不应替代老师做关键判断；

AI 不应记录不必要的敏感个人信息；

AI 不应给学生贴不可解释的负面标签。

第四步，长期记忆：

应记录知识点掌握情况、反复错误类型、练习完成情况、老师偏好和干预记录。

不应记录无关隐私和未经确认的敏感判断。

教育 Agent 的关键不是让 AI 完全接管教学，而是帮助老师形成监督闭环。

3.16 成本、延迟和复杂度

判断是否使用 Agent 时，还要考虑工程成本。

Agent 通常比普通程序更贵。

它可能需要多次模型调用、多次工具调用、更多上下文、更复杂日志、更复杂状态管理和更复杂评估。

例如，同样是生成一封开发信：

LLM 节点可能一次调用即可。

Agent 可能先查客户网站，再总结客户画像，再选择卖点，再生成邮件，再检查风险。质量可能更高，但成本和延迟也更高。

如果用户只需要快速生成一封普通邮件，Agent 可能过度设计。

如果用户要给高价值客户生成高度定制邮件，Agent 的额外成本就值得。

这说明，是否使用 Agent 还取决于任务价值。

高价值、低频、复杂任务，适合 Agent。

低价值、高频、简单任务，不适合 Agent。

例如：

每天为 5 个高价值客户生成定制跟进方案，可以用 Agent。

每天为 5000 条普通消息逐条启动 Agent，不现实。应该用分类模型、规则或批处理。

工程复杂度也要考虑。

Agent 系统需要：

任务状态；

工具注册；

上下文管理；

错误处理；

日志追踪；

权限控制；

人工审批；

评估集；

成本监控。

如果团队没有能力维护这些机制，贸然引入 Agent 可能得不偿失。

所以，Agent 不是为了显得先进，而是要在价值超过复杂度时使用。

3.17 从“自动化阶梯”理解 Agent

可以把任务自动化分成一个阶梯。

第一层：人工操作。

所有事情由人完成，系统只是记录。

第二层：模板辅助。

系统提供模板，人填内容。例如邮件模板、报价模板、报告模板。

第三层：规则自动化。

系统根据固定规则自动处理。例如库存低于阈值提醒。

第四层：LLM 辅助。

系统在某些环节调用模型。例如总结客户回复、生成邮件草稿。

第五层：工作流自动化。

系统控制多步骤流程，确保状态和审批。

第六层：Agent 辅助执行。

Agent 在开放环节中搜索、判断、规划和生成，但高风险动作仍由人确认。

第七层：受控自主 Agent。

Agent 可以在明确权限、预算、目标和停止条件内持续执行。

很多系统不应该直接从第一层跳到第七层。

更合理的路径是逐步升级。

例如外贸获客：

先用模板写开发信；

再用 LLM 生成草稿；

再用表格记录客户；

再让 Agent 辅助找客户；

再加入评分；

再加入审批；

再加入跟进调度；

最后成为长期运行的获客 Agent。

这样做比一开始做“全自动外贸 Agent”更稳。

3.18 本章的核心判断原则

到这里，我们可以把本章总结为几条原则。

第一，确定性优先。

凡是可以用确定性程序解决的问题，不要交给 Agent。计算、校验、去重、状态流转、权限判断，应尽量用规则和代码。

第二，工作流优先。

凡是流程稳定的问题，优先用工作流控制。Agent 可以嵌入节点，但不应该替代流程。

第三，开放判断交给 Agent。

搜索、筛选、理解、生成、复杂判断、失败恢复等开放环节，是 Agent 的价值所在。

第四，高风险动作必须受控。

发送、删除、付款、提交、修改核心数据等动作不能让 Agent 无限制执行。

第五，长期任务需要状态和记忆。

如果任务跨天、跨客户、跨学生、跨项目，就必须设计状态、记忆和调度，而不是只靠上下文。

第六，Agent 的成功经验要沉淀。

不要让 Agent 永远重复思考同样问题。稳定经验应该变成规则、模板、工具和工作流。

第七，是否使用 Agent 取决于价值、风险和复杂度的平衡。

Agent 不是越多越好。真正成熟的系统，会让 Agent 出现在最需要它的位置。

本章总结

本章解决的是 Agent 学习中的一个关键问题：什么时候该用 Agent，什么时候不该用 Agent。

Agent 很强，但不是所有任务都适合 Agent。确定性计算应该用函数，固定步骤应该用脚本或工作流，流程中的理解和生成可以用 LLM 节点，简单实时查询可以用 Tool Calling。只有当任务具有目标开放、多步骤、不确定、需要工具、需要状态或长期记忆等特征时，才真正适合 Agent。

更重要的是，现实系统通常不是“全用 Agent”或“完全不用 Agent”。成熟架构往往是混合的：确定性部分用函数，稳定流程用工作流，开放判断用 Agent，高风险动作由人工确认。

本章还提出了一个重要思想：Agent 可以作为探索层，但成功经验要逐步沉淀为规则、模板、工具和工作流。这样系统才会越来越稳定、便宜和可控，而不是永远依赖模型每次重新思考。

判断边界，是 Agent 工程的起点。一个人是否真正理解 Agent，不在于他能否写出一个循环，而在于他能否面对真实业务时说清楚：哪些部分应该交给 Agent，哪些部分不应该。

下一章开始，我们将进入 Agent 的核心执行机制：Agent Loop。我们会具体分析一个 Agent 如何从观察任务开始，经过思考、计划、行动、反思和停止，形成一个可执行的任务循环。
