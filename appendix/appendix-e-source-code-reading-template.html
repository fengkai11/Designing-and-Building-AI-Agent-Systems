二、第一遍：看产品入口

要回答：用户如何开始一个任务？

检查位置：

README
examples/
apps/
cli/
web/
server/
main.py
index.ts

记录：

- 用户输入在哪里进入系统？
- 输入是自然语言、表单、命令行参数，还是 API？
- 系统创建了什么任务对象？
- 任务如何进入执行队列？

三、第二遍：画执行链路

尝试画出：

User Input
  ↓
Task Creation
  ↓
Context Build
  ↓
Model Call
  ↓
Action Parse
  ↓
Tool Call
  ↓
State Update
  ↓
Stop / Continue / Approval

如果无法画出这条链路，说明还没有真正读懂项目。

四、第三遍：找核心抽象

重点寻找这些对象：

抽象关注问题
Agent它代表角色、任务，还是执行器？
Task任务目标和状态如何表示？
Tool工具如何注册和调用？
Message模型上下文如何表示？
Memory长期信息如何保存？
State执行进度如何保存？
Event日志和事件如何记录？
Approval人工确认如何触发？

五、第四遍：看工具系统

记录：

- 工具在哪里定义？
- 工具如何注册？
- 工具描述是否给模型看？
- 参数如何校验？
- 工具失败如何处理？
- 是否有权限分级？
- 是否有 dry-run？
- 是否有审计日志？

六、第五遍：看上下文构造

记录：

- system prompt 在哪里？
- developer prompt 或运行规则在哪里？
- 历史消息如何裁剪？
- 工具结果如何进入上下文？
- 检索结果如何进入上下文？
- 是否有压缩/摘要机制？
- 是否有 token budget 控制？

七、第六遍：看状态与长任务

记录：

- 任务状态有哪些？
- 是否支持暂停、恢复、取消？
- 是否支持 checkpoint？
- 失败是否可重试？
- 中间产物保存在哪里？
- 多任务是否并发？

八、第七遍：看安全与审批

记录：

- 哪些动作会直接执行？
- 哪些动作需要用户确认？
- 是否限制 shell、文件、网络、邮件等工具？
- 是否防止 prompt injection？
- 是否有敏感信息保护？
- 是否有权限模型？

九、第八遍：看可观测性

记录：

- 是否记录模型输入输出？
- 是否记录工具调用？
- 是否记录 token 和成本？
- 是否支持 trace 回放？
- 是否有 eval？
- 如何定位失败任务？

## 十一、阅读源码时的警惕点

1. 不要把项目目录当成架构本身。
2. 不要因为某个项目没有某模块，就认为该模块不重要。
3. 不要照搬历史包袱。
4. 不要忽略产品场景差异。
5. 不要把闭源产品的外部行为写成确定内部实现。
6. 不要只读 happy path，要读失败、重试、权限和日志。
